# Сессия 4: Передовые модели – LLM, SLM и локальная инференция

## Обзор

Сравните LLM и SLM, оцените компромиссы между локальной и облачной инференцией, а также реализуйте демонстрации, показывающие сценарии EdgeAI с использованием Phi и ONNX Runtime. Мы также рассмотрим Chainlit RAG, варианты инференции с WebGPU и интеграцию Open WebUI.

Ссылки:
- Документация Foundry Local: https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/
- ONNX Runtime GenAI: https://onnxruntime.ai/
- Руководство Open WebUI (чат-приложение с Open WebUI): https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/how-to/how-to-chat-application-with-open-web-ui

## Цели обучения
- Понять компромиссы между LLM и SLM в отношении стоимости, задержки и точности
- Выбрать между локальной и облачной инференцией для конкретных бизнес-задач
- Реализовать небольшой RAG-демо с Chainlit
- Изучить WebGPU для ускорения на стороне браузера
- Подключить Open WebUI к Foundry Local

## Часть 1: LLM vs SLM – Матрица решений

Учитывайте:
- Задержка: SLM на устройстве часто обеспечивает отклик менее секунды
- Стоимость: локальная инференция снижает расходы на облако
- Конфиденциальность: чувствительные данные остаются на устройстве
- Возможности: LLM могут превосходить SLM в сложных задачах
- Надежность: гибридные стратегии снижают риск простоя

## Часть 2: Локальная vs облачная – гибридные подходы

- Локальный подход с облачным резервом для больших/сложных запросов
- Облачный подход с локальным для конфиденциальных данных или офлайн-сценариев
- Маршрутизация по типу задачи (генерация кода – DeepSeek, общий чат – Phi/Qwen)

## Часть 3: RAG чат-приложение с Chainlit (минимальное)

Установите зависимости:
```powershell
python -m venv .venv
.\.venv\Scripts\activate
pip install chainlit requests
```

`app.py`:
```python
import chainlit as cl
import requests

BASE_URL = "http://localhost:8000"
MODEL = "phi-4-mini"

@cl.on_chat_start
async def start():
    await cl.Message(content="Welcome to Local RAG Chat!").send()

@cl.on_message
async def main(message: cl.Message):
    # Basic passthrough to Foundry Local (extend with retrieval)
    payload = {
        "model": MODEL,
        "messages": [{"role": "user", "content": message.content}],
        "max_tokens": 300
    }
    r = requests.post(f"{BASE_URL}/v1/chat/completions", json=payload, timeout=60)
    r.raise_for_status()
    content = r.json()["choices"][0]["message"]["content"]
    await cl.Message(content=content).send()
```

Запустите:
```powershell
# Ensure a model is running
foundry model run phi-4-mini

# Launch Chainlit
chainlit run app.py -w
```

Расширьте: добавьте простой механизм поиска (локальные файлы) и добавьте найденный контекст к пользовательскому запросу.

## Часть 4: Инференция с WebGPU (важное)

Запускайте небольшие модели прямо в браузере с использованием WebGPU. Это идеально для демонстраций с приоритетом конфиденциальности и без необходимости установки. Ниже приведен минимальный пошаговый пример с использованием ONNX Runtime Web и провайдера выполнения WebGPU.

1) Проверьте поддержку WebGPU
- Браузеры на базе Chromium: chrome://gpu → убедитесь, что “WebGPU” включен
- Программная проверка (мы также проверим в коде): `if (!('gpu' in navigator)) { /* no WebGPU */ }`

2) Создайте минимальный проект
Создайте папку и два файла: `index.html` и `main.js`.

`index.html`
```html
<!doctype html>
<html>
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <title>WebGPU + ONNX Runtime Web (Demo)</title>
    <!-- Load ONNX Runtime Web (WebGPU build) from CDN -->
    <script src="https://cdn.jsdelivr.net/npm/onnxruntime-web/dist/ort.webgpu.min.js"></script>
    <style>
      body { font-family: system-ui, sans-serif; margin: 2rem; }
      pre { background: #f5f5f5; padding: 1rem; overflow:auto; }
    </style>
  </head>
  <body>
    <h1>WebGPU + ONNX Runtime Web</h1>
    <p id="status">Initializing…</p>
    <pre id="output"></pre>
    <script type="module" src="./main.js"></script>
  </body>
</html>
```

`main.js`
```javascript
const statusEl = document.getElementById('status');
const outputEl = document.getElementById('output');

function log(msg) {
  outputEl.textContent += `${msg}\n`;
}

(async () => {
  try {
    if (!('gpu' in navigator)) {
      statusEl.textContent = 'WebGPU not available in this browser.';
      return;
    }
    statusEl.textContent = 'WebGPU detected. Loading model…';

    // Use a tiny identity model (or any small ONNX) hosted remotely.
    // For demo, we synthesize a trivial input and expect the same output shape.
    // Replace with your own model URL if desired.
    const modelUrl = 'https://huggingface.co/onnx/models/resolve/main/vision/classification/mnist-12/mnist-12.onnx?download=true';

    // Create a WebGPU session
    const session = await ort.InferenceSession.create(modelUrl, {
      executionProviders: ['webgpu']
    });
    log('Session created with WebGPU.');

    // Create a dummy input of the correct shape for MNIST (1x1x28x28)
    const inputData = new Float32Array(1 * 1 * 28 * 28);
    // Minimal non-zero to avoid NaNs in some models
    inputData[0] = 1.0;
    const input = new ort.Tensor('float32', inputData, [1, 1, 28, 28]);

    // The MNIST model input name is typically 'Input3' or similar; we can inspect session.inputNames
    const feeds = {};
    const inputName = session.inputNames[0];
    feeds[inputName] = input;

    const results = await session.run(feeds);
    const outputName = session.outputNames[0];
    const output = results[outputName]; // Float32Array logits

    statusEl.textContent = 'Inference complete.';
    log(`Input name: ${inputName}`);
    log(`Output name: ${outputName}`);
    log(`Output length: ${output.data.length}`);

    // Argmax to get predicted digit
    let bestIdx = 0; let bestVal = -Infinity;
    for (let i = 0; i < output.data.length; i++) {
      if (output.data[i] > bestVal) { bestVal = output.data[i]; bestIdx = i; }
    }
    log(`Predicted class (argmax): ${bestIdx}`);
  } catch (e) {
    statusEl.textContent = 'Error during inference.';
    log(e?.stack || String(e));
  }
})();
```

3) Запустите локально (Windows cmd.exe)
Используйте простой статический сервер, чтобы браузер мог получить модель.

```cmd
REM In cmd.exe
mkdir webgpu-demo
cd webgpu-demo
REM Save index.html and main.js here

REM Option A: Python (if installed)
py -m http.server 5173

REM Option B: Node serve (requires Node.js)
REM npx serve -p 5173 .
```

Откройте http://localhost:5173 в браузере. Вы должны увидеть логи инициализации, создание сессии с WebGPU и предсказание argmax.

4) Устранение неполадок
- Если WebGPU недоступен: обновите Chrome/Edge, убедитесь, что драйверы GPU актуальны, затем проверьте chrome://flags на наличие “Enable WebGPU”.
- Если возникают ошибки CORS или fetch: убедитесь, что файлы обслуживаются через http:// (а не file://), и URL модели разрешает запросы из других источников.
- Резервный вариант на CPU: измените `executionProviders: ['wasm']`, чтобы проверить базовое поведение.

5) Следующие шаги
- Замените модель на специализированную ONNX (например, классификация изображений или небольшая текстовая модель).
- Добавьте логику предварительной/последующей обработки для реальных входных данных.
- Для больших моделей или производственной задержки предпочтите Foundry Local или ONNX Runtime Server.

## Часть 5: Open WebUI + Foundry Local (пошагово)

Это соединяет Open WebUI с совместимым с OpenAI API Foundry Local для локального интерфейса чата.

1) Предварительные условия
- Установлен и работает Foundry Local (`foundry --version`)
- Одна модель готова к запуску локально (например, `phi-4-mini`)
- Установлен Docker Desktop (рекомендуется для Open WebUI)

2) Запустите модель с Foundry Local
```powershell
foundry model run phi-4-mini
```
Это откроет API, совместимый с OpenAI, по адресу `http://localhost:8000`.

3) Запустите Open WebUI (Docker)
```powershell
# Pull and run Open WebUI
docker pull ghcr.io/open-webui/open-webui:main

docker run -d --name open-webui -p 3000:8080 ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
Примечания:
- В Windows `host.docker.internal` позволяет контейнеру обращаться к вашему хосту по адресу `localhost`.
- Мы устанавливаем `OPENAI_API_BASE_URL` на конечную точку Foundry Local и фиктивный `OPENAI_API_KEY`.

4) Настройка через интерфейс Open WebUI (альтернатива)
- Перейдите на http://localhost:3000
- Завершите начальную настройку (администратор)
- Перейдите в Настройки → Модели/Провайдеры
- Установите базовый URL: `http://host.docker.internal:8000/v1`
- Установите API-ключ: `local-key` (заглушка)
- Сохраните

5) Запустите тестовый запрос
- В чате Open WebUI выберите или введите имя модели `phi-4-mini`
- Запрос: “Перечислите пять преимуществ инференции на устройстве.”
- Вы должны увидеть ответ, переданный от вашей локальной модели

6) Устранение неполадок
```powershell
# Verify Foundry Local is serving and responding
curl http://localhost:8000/v1/models

# Check Docker logs for Open WebUI
docker logs -f open-webui

# If you get ECONNREFUSED, ensure the model is running and port 8000 is open
# If the model name doesn't appear, try typing it manually in Open WebUI
```

7) Дополнительно: сохранение данных Open WebUI
```powershell
# Stop and remove the container
docker stop open-webui
docker rm open-webui

# Re-run with a volume to persist data
mkdir %USERPROFILE%\openwebui-data

docker run -d --name open-webui -p 3000:8080 ^
  -v %USERPROFILE%\openwebui-data:/app/backend/data ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
