## บทนำ

ยินดีต้อนรับสู่คู่มือการศึกษา EdgeAI สำหรับผู้เริ่มต้น! เอกสารนี้ออกแบบมาเพื่อช่วยให้คุณสำรวจเนื้อหาหลักสูตรได้อย่างมีประสิทธิภาพและเพิ่มประสบการณ์การเรียนรู้ของคุณให้สูงสุด โดยมีเส้นทางการเรียนรู้ที่มีโครงสร้าง ตารางการศึกษาแนะนำ สรุปแนวคิดสำคัญ และทรัพยากรเสริมเพื่อเพิ่มความเข้าใจในเทคโนโลยี EdgeAI

นี่คือหลักสูตรที่กระชับ ใช้เวลา 20 ชั่วโมง ซึ่งมอบความรู้สำคัญเกี่ยวกับ EdgeAI ในรูปแบบที่ประหยัดเวลา เหมาะสำหรับมืออาชีพและนักเรียนที่มีเวลาจำกัดและต้องการพัฒนาทักษะในสาขาที่กำลังเติบโตนี้อย่างรวดเร็ว

## ภาพรวมของหลักสูตร

หลักสูตรนี้แบ่งออกเป็นเจ็ดโมดูลที่ครอบคลุม:

1. **พื้นฐานและการเปลี่ยนแปลงของ EdgeAI** - ทำความเข้าใจแนวคิดหลักและการเปลี่ยนแปลงทางเทคโนโลยี
2. **พื้นฐานของ Small Language Model (SLM)** - สำรวจครอบครัว SLM ต่าง ๆ และสถาปัตยกรรมของพวกเขา
3. **การปรับใช้ Small Language Model** - การปรับใช้ในเชิงปฏิบัติ
4. **การแปลงรูปแบบโมเดลและการลดขนาด** - การปรับแต่งขั้นสูงด้วย 6 เฟรมเวิร์ก รวมถึง OpenVINO
5. **SLMOps - การดำเนินงานของ Small Language Model** - การจัดการวงจรการผลิตและการปรับใช้
6. **ระบบ SLM Agentic** - AI agents, การเรียกฟังก์ชัน และ Model Context Protocol
7. **ตัวอย่างการใช้งาน EdgeAI** - AI Toolkit, การพัฒนาบน Windows และการใช้งานเฉพาะแพลตฟอร์ม
8. **Microsoft Foundry Local – ชุดเครื่องมือสำหรับนักพัฒนาแบบครบวงจร** - การพัฒนาแบบ local-first พร้อมการผสาน Azure แบบไฮบริด (โมดูล 08)

## วิธีใช้คู่มือการศึกษา

- **การเรียนรู้แบบก้าวหน้า**: เรียนตามลำดับโมดูลเพื่อประสบการณ์การเรียนรู้ที่ต่อเนื่อง
- **จุดตรวจสอบความรู้**: ใช้คำถามประเมินตนเองหลังแต่ละส่วน
- **การฝึกปฏิบัติ**: ทำแบบฝึกหัดที่แนะนำเพื่อเสริมสร้างแนวคิดเชิงทฤษฎี
- **ทรัพยากรเสริม**: สำรวจเนื้อหาเพิ่มเติมสำหรับหัวข้อที่คุณสนใจมากที่สุด

## โมดูล 1: พื้นฐานและการเปลี่ยนแปลงของ EdgeAI

### วัตถุประสงค์การเรียนรู้สำคัญ

- เข้าใจความแตกต่างระหว่าง AI บนคลาวด์และ AI บน Edge
- เชี่ยวชาญเทคนิคการปรับแต่งหลักสำหรับสภาพแวดล้อมที่มีทรัพยากรจำกัด
- วิเคราะห์การใช้งานจริงของเทคโนโลยี EdgeAI
- ตั้งค่าสภาพแวดล้อมการพัฒนาสำหรับโปรเจกต์ EdgeAI

### จุดเน้นการศึกษา

#### ส่วนที่ 1: พื้นฐาน EdgeAI
- **แนวคิดสำคัญ**: 
  - การเปรียบเทียบระหว่าง Edge และ Cloud computing
  - เทคนิคการลดขนาดโมเดล
  - ตัวเลือกการเร่งความเร็วฮาร์ดแวร์ (NPUs, GPUs, CPUs)
  - ข้อดีด้านความเป็นส่วนตัวและความปลอดภัย

- **ทรัพยากรเสริม**:
  - [TensorFlow Lite Documentation](https://www.tensorflow.org/lite)
  - [ONNX Runtime GitHub](https://github.com/microsoft/onnxruntime)
  - [Edge Impulse Documentation](https://docs.edgeimpulse.com)

#### ส่วนที่ 2: กรณีศึกษาในโลกจริง
- **แนวคิดสำคัญ**: 
  - ระบบโมเดล Microsoft Phi & Mu
  - การใช้งานจริงในอุตสาหกรรมต่าง ๆ
  - ข้อควรพิจารณาในการปรับใช้

#### ส่วนที่ 3: คู่มือการใช้งานในเชิงปฏิบัติ
- **แนวคิดสำคัญ**: 
  - การตั้งค่าสภาพแวดล้อมการพัฒนา
  - เครื่องมือการลดขนาดและการปรับแต่ง
  - วิธีการประเมินการใช้งาน EdgeAI

#### ส่วนที่ 4: ฮาร์ดแวร์สำหรับการปรับใช้ Edge
- **แนวคิดสำคัญ**: 
  - การเปรียบเทียบแพลตฟอร์มฮาร์ดแวร์
  - กลยุทธ์การปรับแต่งสำหรับฮาร์ดแวร์เฉพาะ
  - ข้อควรพิจารณาในการปรับใช้

### คำถามประเมินตนเอง

1. เปรียบเทียบและอธิบายความแตกต่างระหว่าง AI บนคลาวด์และ AI บน Edge
2. อธิบายเทคนิคสำคัญสามประการสำหรับการปรับแต่งโมเดลเพื่อการปรับใช้บน Edge
3. ข้อดีหลักของการรันโมเดล AI บน Edge คืออะไร?
4. อธิบายกระบวนการลดขนาดโมเดลและผลกระทบต่อประสิทธิภาพ
5. อธิบายว่าตัวเร่งความเร็วฮาร์ดแวร์ (NPUs, GPUs, CPUs) มีผลต่อการปรับใช้ EdgeAI อย่างไร

### แบบฝึกหัดปฏิบัติ

1. **การตั้งค่าสภาพแวดล้อมอย่างรวดเร็ว**: ตั้งค่าสภาพแวดล้อมการพัฒนาขั้นต่ำด้วยแพ็กเกจที่จำเป็น (30 นาที)
2. **การสำรวจโมเดล**: ดาวน์โหลดและตรวจสอบโมเดลภาษาเล็กที่ผ่านการฝึกมาแล้ว (1 ชั่วโมง)
3. **การลดขนาดพื้นฐาน**: ทดลองลดขนาดโมเดลเล็ก (1 ชั่วโมง)

## โมดูล 2: พื้นฐานของ Small Language Model

### วัตถุประสงค์การเรียนรู้สำคัญ

- เข้าใจหลักการสถาปัตยกรรมของครอบครัว SLM ต่าง ๆ
- เปรียบเทียบความสามารถของโมเดลในระดับพารามิเตอร์ต่าง ๆ
- ประเมินโมเดลตามประสิทธิภาพ ความสามารถ และข้อกำหนดการปรับใช้
- ระบุกรณีการใช้งานที่เหมาะสมสำหรับครอบครัวโมเดลต่าง ๆ

### จุดเน้นการศึกษา

#### ส่วนที่ 1: ครอบครัวโมเดล Microsoft Phi
- **แนวคิดสำคัญ**: 
  - วิวัฒนาการของปรัชญาการออกแบบ
  - สถาปัตยกรรมที่เน้นประสิทธิภาพ
  - ความสามารถเฉพาะทาง

#### ส่วนที่ 2: ครอบครัว Qwen
- **แนวคิดสำคัญ**: 
  - การมีส่วนร่วมในโอเพ่นซอร์ส
  - ตัวเลือกการปรับใช้ที่ปรับขนาดได้
  - สถาปัตยกรรมการให้เหตุผลขั้นสูง

#### ส่วนที่ 3: ครอบครัว Gemma
- **แนวคิดสำคัญ**: 
  - นวัตกรรมที่ขับเคลื่อนด้วยการวิจัย
  - ความสามารถแบบมัลติโหมด
  - การปรับแต่งสำหรับมือถือ

#### ส่วนที่ 4: ครอบครัว BitNET
- **แนวคิดสำคัญ**: 
  - เทคโนโลยีการลดขนาด 1-bit
  - เฟรมเวิร์กการปรับแต่งการอนุมาน
  - ข้อพิจารณาด้านความยั่งยืน

#### ส่วนที่ 5: โมเดล Microsoft Mu
- **แนวคิดสำคัญ**: 
  - สถาปัตยกรรมที่เน้นอุปกรณ์
  - การผสานระบบกับ Windows
  - การดำเนินงานที่รักษาความเป็นส่วนตัว

#### ส่วนที่ 6: Phi-Silica
- **แนวคิดสำคัญ**: 
  - สถาปัตยกรรมที่ปรับแต่งสำหรับ NPU
  - ตัวชี้วัดประสิทธิภาพ
  - การผสานสำหรับนักพัฒนา

### คำถามประเมินตนเอง

1. เปรียบเทียบแนวทางสถาปัตยกรรมของครอบครัวโมเดล Phi และ Qwen
2. อธิบายว่าเทคโนโลยีการลดขนาดของ BitNET แตกต่างจากการลดขนาดแบบดั้งเดิมอย่างไร
3. ข้อดีเฉพาะของโมเดล Mu สำหรับการผสาน Windows คืออะไร?
4. อธิบายว่า Phi-Silica ใช้ฮาร์ดแวร์ NPU เพื่อเพิ่มประสิทธิภาพอย่างไร
5. สำหรับแอปพลิเคชันมือถือที่มีการเชื่อมต่อจำกัด ครอบครัวโมเดลใดที่เหมาะสมที่สุดและเพราะเหตุใด

### แบบฝึกหัดปฏิบัติ

1. **การเปรียบเทียบโมเดล**: ทดสอบประสิทธิภาพของโมเดล SLM สองตัว (1 ชั่วโมง)
2. **การสร้างข้อความพื้นฐาน**: การใช้งานพื้นฐานของการสร้างข้อความด้วยโมเดลเล็ก (1 ชั่วโมง)
3. **การปรับแต่งอย่างรวดเร็ว**: ใช้เทคนิคการปรับแต่งหนึ่งอย่างเพื่อเพิ่มความเร็วในการอนุมาน (1 ชั่วโมง)

## โมดูล 3: การปรับใช้ Small Language Model

### วัตถุประสงค์การเรียนรู้สำคัญ

- เลือกโมเดลที่เหมาะสมตามข้อจำกัดการปรับใช้
- เชี่ยวชาญเทคนิคการปรับแต่งสำหรับสถานการณ์การปรับใช้ต่าง ๆ
- ใช้งาน SLM ในทั้งสภาพแวดล้อมแบบ local และ cloud
- ออกแบบการตั้งค่าที่พร้อมสำหรับการผลิตสำหรับแอปพลิเคชัน EdgeAI

### จุดเน้นการศึกษา

#### ส่วนที่ 1: การเรียนรู้ขั้นสูงของ SLM
- **แนวคิดสำคัญ**: 
  - เฟรมเวิร์กการจำแนกพารามิเตอร์
  - เทคนิคการปรับแต่งขั้นสูง
  - กลยุทธ์การได้มาของโมเดล

#### ส่วนที่ 2: การปรับใช้ในสภาพแวดล้อมแบบ local
- **แนวคิดสำคัญ**: 
  - การปรับใช้แพลตฟอร์ม Ollama
  - โซลูชัน Microsoft Foundry local
  - การวิเคราะห์เปรียบเทียบเฟรมเวิร์ก

#### ส่วนที่ 3: การปรับใช้แบบ cloud ด้วย container
- **แนวคิดสำคัญ**: 
  - การอนุมานประสิทธิภาพสูงด้วย vLLM
  - การจัดการ container
  - การใช้งาน ONNX Runtime

### คำถามประเมินตนเอง

1. ปัจจัยใดที่ควรพิจารณาเมื่อเลือกการปรับใช้แบบ local และแบบ cloud?
2. เปรียบเทียบ Ollama และ Microsoft Foundry Local ในฐานะตัวเลือกการปรับใช้
3. อธิบายข้อดีของการใช้ container ในการปรับใช้ SLM
4. ตัวชี้วัดประสิทธิภาพหลักที่ควรติดตามสำหรับ SLM ที่ปรับใช้บน Edge คืออะไร?
5. อธิบายขั้นตอนการปรับใช้ตั้งแต่การเลือกโมเดลจนถึงการใช้งานในผลิตภัณฑ์

### แบบฝึกหัด
- OpenVINO GenAI สำหรับการใช้งาน LLM

#### ส่วนที่ 5: Apple MLX Framework
- **แนวคิดสำคัญ**: 
  - การปรับแต่งสำหรับ Apple Silicon
  - สถาปัตยกรรมหน่วยความจำแบบรวม
  - ความสามารถในการปรับแต่ง LoRA

#### ส่วนที่ 6: การสังเคราะห์เวิร์กโฟลว์ Edge AI
- **แนวคิดสำคัญ**: 
  - สถาปัตยกรรมเวิร์กโฟลว์แบบรวม
  - การตัดสินใจเลือก Framework ด้วย Decision Trees
  - การตรวจสอบความพร้อมสำหรับการผลิต
  - กลยุทธ์เพื่อรองรับอนาคต

### คำถามประเมินตนเอง

1. เปรียบเทียบกลยุทธ์การลดความแม่นยำในระดับต่าง ๆ (1-bit ถึง 8-bit)
2. อธิบายข้อดีของรูปแบบ GGUF สำหรับการใช้งาน Edge
3. การปรับแต่งฮาร์ดแวร์ใน Microsoft Olive ช่วยเพิ่มประสิทธิภาพการใช้งานได้อย่างไร?
4. ประโยชน์สำคัญของ OpenVINO's NNCF สำหรับการบีบอัดโมเดลคืออะไร?
5. Apple MLX ใช้สถาปัตยกรรมหน่วยความจำแบบรวมเพื่อการปรับแต่งอย่างไร?
6. การสังเคราะห์เวิร์กโฟลว์ช่วยในการเลือก Framework ที่เหมาะสมได้อย่างไร?

### แบบฝึกหัดปฏิบัติ

1. **การลดความแม่นยำของโมเดล**: ใช้ระดับการลดความแม่นยำต่าง ๆ กับโมเดลและเปรียบเทียบผลลัพธ์ (1 ชั่วโมง)
2. **การปรับแต่ง OpenVINO**: ใช้ NNCF เพื่อบีบอัดโมเดลสำหรับฮาร์ดแวร์ Intel (1 ชั่วโมง)
3. **การเปรียบเทียบ Framework**: ทดสอบโมเดลเดียวกันใน Framework การปรับแต่งสามแบบ (1 ชั่วโมง)
4. **การวัดประสิทธิภาพ**: วัดผลกระทบของการปรับแต่งต่อความเร็วในการประมวลผลและการใช้หน่วยความจำ (1 ชั่วโมง)

## โมดูล 5: SLMOps - การดำเนินงานโมเดลภาษาขนาดเล็ก

### วัตถุประสงค์การเรียนรู้หลัก

- เข้าใจหลักการจัดการวงจรชีวิตของ SLMOps
- เชี่ยวชาญเทคนิคการลดขนาดและการปรับแต่งสำหรับการใช้งาน Edge
- ใช้กลยุทธ์การใช้งานในระดับการผลิตพร้อมการตรวจสอบ
- สร้างเวิร์กโฟลว์การดำเนินงานและการบำรุงรักษา SLM ระดับองค์กร

### พื้นที่การศึกษา

#### ส่วนที่ 1: บทนำสู่ SLMOps
- **แนวคิดสำคัญ**: 
  - การเปลี่ยนแปลงแนวคิด SLMOps ในการดำเนินงาน AI
  - สถาปัตยกรรมที่คุ้มค่าและเน้นความเป็นส่วนตัว
  - ผลกระทบเชิงกลยุทธ์ต่อธุรกิจและข้อได้เปรียบในการแข่งขัน

#### ส่วนที่ 2: การลดขนาดโมเดล
- **แนวคิดสำคัญ**: 
  - เทคนิคการถ่ายโอนความรู้
  - การดำเนินการกระบวนการลดขนาดสองขั้นตอน
  - เวิร์กโฟลว์การลดขนาดใน Azure ML

#### ส่วนที่ 3: กลยุทธ์การปรับแต่ง
- **แนวคิดสำคัญ**: 
  - การปรับแต่งที่มีประสิทธิภาพต่อพารามิเตอร์ (PEFT)
  - วิธีการขั้นสูง LoRA และ QLoRA
  - การฝึกอบรมหลายตัวแปลงและการปรับแต่งไฮเปอร์พารามิเตอร์

#### ส่วนที่ 4: การใช้งานในระดับการผลิต
- **แนวคิดสำคัญ**: 
  - การแปลงและลดความแม่นยำของโมเดลสำหรับการผลิต
  - การกำหนดค่าการใช้งาน Foundry Local
  - การวัดประสิทธิภาพและการตรวจสอบคุณภาพ

### คำถามประเมินตนเอง

1. SLMOps แตกต่างจาก MLOps แบบดั้งเดิมอย่างไร?
2. อธิบายประโยชน์ของการลดขนาดโมเดลสำหรับการใช้งาน Edge
3. ข้อควรพิจารณาสำคัญสำหรับการปรับแต่ง SLM ในสภาพแวดล้อมที่มีทรัพยากรจำกัดคืออะไร?
4. อธิบายกระบวนการใช้งานในระดับการผลิตสำหรับแอปพลิเคชัน AI Edge

### แบบฝึกหัดปฏิบัติ

1. **การลดขนาดพื้นฐาน**: สร้างโมเดลขนาดเล็กจากโมเดลครูขนาดใหญ่ (1 ชั่วโมง)
2. **การทดลองปรับแต่ง**: ปรับแต่งโมเดลสำหรับโดเมนเฉพาะ (1 ชั่วโมง)
3. **เวิร์กโฟลว์การใช้งาน**: ตั้งค่าเวิร์กโฟลว์ CI/CD พื้นฐานสำหรับการใช้งานโมเดล (1 ชั่วโมง)

## โมดูล 6: ระบบ SLM Agentic - AI Agents และการเรียกฟังก์ชัน

### วัตถุประสงค์การเรียนรู้หลัก

- สร้าง AI Agents อัจฉริยะสำหรับสภาพแวดล้อม Edge โดยใช้ Small Language Models
- ใช้ความสามารถในการเรียกฟังก์ชันด้วยเวิร์กโฟลว์ที่เป็นระบบ
- เชี่ยวชาญการรวม Model Context Protocol (MCP) เพื่อการโต้ตอบเครื่องมือที่ได้มาตรฐาน
- สร้างระบบ Agentic ที่ซับซ้อนด้วยการแทรกแซงของมนุษย์น้อยที่สุด

### พื้นที่การศึกษา

#### ส่วนที่ 1: AI Agents และพื้นฐาน SLM
- **แนวคิดสำคัญ**: 
  - กรอบการจำแนกประเภท Agent (Reflex, Model-based, Goal-based, Learning Agents)
  - การวิเคราะห์ข้อแลกเปลี่ยนระหว่าง SLM และ LLM
  - รูปแบบการออกแบบ Agent เฉพาะ Edge
  - การปรับทรัพยากรสำหรับ Agent

#### ส่วนที่ 2: การเรียกฟังก์ชันใน Small Language Models
- **แนวคิดสำคัญ**: 
  - การดำเนินการเวิร์กโฟลว์อย่างเป็นระบบ (การตรวจจับเจตนา, JSON Output, การดำเนินการภายนอก)
  - การใช้งานเฉพาะแพลตฟอร์ม (Phi-4-mini, โมเดล Qwen ที่เลือก, Microsoft Foundry Local)
  - ตัวอย่างขั้นสูง (การทำงานร่วมกันของหลาย Agent, การเลือกเครื่องมือแบบไดนามิก)
  - ข้อควรพิจารณาสำหรับการผลิต (การจำกัดอัตรา, การบันทึกการตรวจสอบ, มาตรการรักษาความปลอดภัย)

#### ส่วนที่ 3: การรวม Model Context Protocol (MCP)
- **แนวคิดสำคัญ**: 
  - สถาปัตยกรรมโปรโตคอลและการออกแบบระบบแบบเลเยอร์
  - การสนับสนุนหลาย Backend (Ollama สำหรับการพัฒนา, vLLM สำหรับการผลิต)
  - โปรโตคอลการเชื่อมต่อ (โหมด STDIO และ SSE)
  - การใช้งานในโลกจริง (การทำงานอัตโนมัติบนเว็บ, การประมวลผลข้อมูล, การรวม API)

### คำถามประเมินตนเอง

1. ข้อควรพิจารณาสำคัญสำหรับสถาปัตยกรรม AI Agents ใน Edge คืออะไร?
2. การเรียกฟังก์ชันช่วยเพิ่มความสามารถของ Agent ได้อย่างไร?
3. อธิบายบทบาทของ Model Context Protocol ในการสื่อสารของ Agent

### แบบฝึกหัดปฏิบัติ

1. **Agent แบบง่าย**: สร้าง AI Agent พื้นฐานพร้อมการเรียกฟังก์ชัน (1 ชั่วโมง)
2. **การรวม MCP**: ใช้ MCP ในแอปพลิเคชัน Agent (30 นาที)

## โมดูล 7: ตัวอย่างการใช้งาน EdgeAI

### วัตถุประสงค์การเรียนรู้หลัก

- เชี่ยวชาญ AI Toolkit สำหรับ Visual Studio Code เพื่อเวิร์กโฟลว์การพัฒนา EdgeAI ที่ครอบคลุม
- ได้รับความเชี่ยวชาญในแพลตฟอร์ม Windows AI Foundry และกลยุทธ์การปรับแต่ง NPU
- ใช้ EdgeAI ในหลายแพลตฟอร์มฮาร์ดแวร์และสถานการณ์การใช้งาน
- สร้างแอปพลิเคชัน EdgeAI ที่พร้อมสำหรับการผลิตด้วยการปรับแต่งเฉพาะแพลตฟอร์ม

### พื้นที่การศึกษา

#### ส่วนที่ 1: AI Toolkit สำหรับ Visual Studio Code
- **แนวคิดสำคัญ**: 
  - สภาพแวดล้อมการพัฒนา Edge AI ที่ครอบคลุมภายใน VS Code
  - แคตตาล็อกโมเดลและการค้นพบสำหรับการใช้งาน Edge
  - เวิร์กโฟลว์การทดสอบในเครื่อง, การปรับแต่ง, และการพัฒนา Agent
  - การตรวจสอบประสิทธิภาพและการประเมินสำหรับสถานการณ์ Edge

#### ส่วนที่ 2: คู่มือการพัฒนา Windows EdgeAI
- **แนวคิดสำคัญ**: 
  - ภาพรวมแพลตฟอร์ม Windows AI Foundry

  - Computer Vision APIs สำหรับการประมวลผลภาพและ OCR
  - Foundry Local CLI สำหรับการพัฒนาและการทดสอบในเครื่อง

#### ส่วนที่ 3: การใช้งานเฉพาะแพลตฟอร์ม
- **แนวคิดสำคัญ**: 
  - การใช้งาน NVIDIA Jetson Orin Nano (ประสิทธิภาพ AI 67 TOPS)
  - แอปพลิเคชันมือถือด้วย .NET MAUI และ ONNX Runtime GenAI
  - โซลูชัน Azure EdgeAI ด้วยสถาปัตยกรรมไฮบริด Cloud-Edge
  - การปรับแต่ง Windows ML ด้วยการสนับสนุนฮาร์ดแวร์แบบสากล
  - แอปพลิเคชัน Foundry Local พร้อมการใช้งาน RAG ที่เน้นความเป็นส่วนตัว

### คำถามประเมินตนเอง

1. AI Toolkit ช่วยปรับปรุงเวิร์กโฟลว์การพัฒนา EdgeAI ได้อย่างไร?
2. เปรียบเทียบกลยุทธ์การใช้งานในแพลตฟอร์มฮาร์ดแวร์ต่าง ๆ
3. ข้อดีของ Windows AI Foundry สำหรับการพัฒนา Edge คืออะไร?
4. บทบาทของการปรับแต่ง NPU ในแอปพลิเคชัน Edge AI สมัยใหม่คืออะไร?
5. Phi Silica API ใช้ฮาร์ดแวร์ NPU เพื่อการปรับแต่งประสิทธิภาพอย่างไร?
6. เปรียบเทียบข้อดีของการใช้งานในเครื่องกับการใช้งานบน Cloud สำหรับแอปพลิเคชันที่เน้นความเป็นส่วนตัว

### แบบฝึกหัดปฏิบัติ

### วัตถุประสงค์การเรียนรู้หลัก

- ติดตั้งและกำหนดค่า Foundry Local พร้อมการรวม SDK สมัยใหม่
- ใช้ระบบหลาย Agent ขั้นสูงด้วยรูปแบบ Coordinator
- สร้างตัวเลือกโมเดลอัจฉริยะด้วยการเลือกอัตโนมัติตามงาน
- ใช้งานโซลูชัน AI ที่พร้อมสำหรับการผลิตพร้อมการตรวจสอบที่ครอบคลุม
- รวมเข้ากับ Azure AI Foundry สำหรับสถานการณ์การใช้งานแบบไฮบริด
- เชี่ยวชาญรูปแบบ SDK สมัยใหม่ด้วย FoundryLocalManager และ OpenAI Client

### พื้นที่การศึกษา

#### ส่วนที่ 1: การติดตั้งและกำหนดค่าที่ทันสมัย
- **แนวคิดสำคัญ**: 
  - การรวม SDK FoundryLocalManager
  - การค้นพบบริการอัตโนมัติและการตรวจสอบสุขภาพ
  - รูปแบบการกำหนดค่าตามสภาพแวดล้อม
  - ข้อควรพิจารณาสำหรับการใช้งานในระดับการผลิต

#### ส่วนที่ 2: ระบบหลาย Agent ขั้นสูง
- **แนวคิดสำคัญ**: 
  - รูปแบบ Coordinator พร้อม Agent เฉพาะทาง
  - การดึงข้อมูล, การให้เหตุผล, และการดำเนินการเฉพาะทางของ Agent
  - กลไก Feedback Loop สำหรับการปรับปรุง
  - การตรวจสอบประสิทธิภาพและการติดตามสถิติ

#### ส่วนที่ 3: การเลือกโมเดลอัจฉริยะ
- **แนวคิดสำคัญ**: 
  - อัลกอริทึมการเลือกโมเดลตามคำสำคัญ
  - การสนับสนุนโมเดลหลายตัว (ทั่วไป, การให้เหตุผล, โค้ด, สร้างสรรค์)
  - การกำหนดค่าตัวแปรสภาพแวดล้อมเพื่อความยืดหยุ่น
  - การตรวจสอบสุขภาพบริการและการจัดการข้อผิดพลาด

#### ส่วนที่ 4: การใช้งานที่พร้อมสำหรับการผลิต
- **แนวคิดสำคัญ**: 
  - การจัดการข้อผิดพลาดและกลไกสำรองที่ครอบคลุม
  - การตรวจสอบคำขอและการติดตามประสิทธิภาพ
  - ตัวอย่าง Jupyter Notebook แบบโต้ตอบพร้อมการวัดประสิทธิภาพ
  - รูปแบบการรวมเข้ากับแอปพลิเคชันที่มีอยู่

### คำถามประเมินตนเอง

1. วิธีการ FoundryLocalManager สมัยใหม่แตกต่างจากการเรียก REST แบบแมนนวลอย่างไร?
2. อธิบายรูปแบบ Coordinator และวิธีการจัดการ Agent เฉพาะทาง
3. Router อัจฉริยะเลือกโมเดลที่เหมาะสมตามเนื้อหาคำค้นหาได้อย่างไร?
4. ส่วนประกอบสำคัญของระบบ AI Agent ที่พร้อมสำหรับการผลิตคืออะไร?
5. คุณจะดำเนินการตรวจสอบสุขภาพที่ครอบคลุมสำหรับบริการ Foundry Local ได้อย่างไร?
6. เปรียบเทียบข้อดีของวิธีการที่ปรับปรุงใหม่กับรูปแบบการใช้งานแบบดั้งเดิม

### แบบฝึกหัดปฏิบัติ

1. **การตั้งค่า SDK สมัยใหม่**: ตั้งค่า FoundryLocalManager พร้อมการค้นพบบริการอัตโนมัติ (30 นาที)
2. **ระบบหลาย Agent**: ใช้ Coordinator ขั้นสูงพร้อม Agent เฉพาะทาง (30 นาที)
3. **การเลือกอัจฉริยะ**: ทดสอบ Router โมเดลด้วยคำค้นหาประเภทต่าง ๆ (30 นาที)
4. **การสำรวจแบบโต้ตอบ**: ใช้ Jupyter Notebook เพื่อสำรวจฟีเจอร์ขั้นสูง (45 นาที)
5. **การใช้งานในระดับการผลิต**: ใช้รูปแบบการตรวจสอบและการจัดการข้อผิดพลาด (30 นาที)
6. **การรวมแบบไฮบริด**: ตั้งค่ากลไกสำรอง Azure AI Foundry (30 นาที)

## ไอเดียโครงการขนาดเล็ก

ลองทำหนึ่งในโครงการเหล่านี้เพื่อฝึกฝนแนวคิด EdgeAI (แต่ละโครงการใช้เวลา 2-4 ชั่วโมง):

### โครงการระดับเริ่มต้น (2-3 ชั่วโมงต่อโครงการ)
1. **Edge Text Assistant**: สร้างเครื่องมือเติมข้อความแบบออฟไลน์ง่ายๆ โดยใช้โมเดลภาษาขนาดเล็ก
2. **Model Comparison Dashboard**: สร้างการแสดงผลพื้นฐานของตัวชี้วัดประสิทธิภาพระหว่าง SLMs ต่างๆ
3. **Optimization Experiment**: วัดผลกระทบของระดับการปรับแต่งต่างๆ บนโมเดลพื้นฐานเดียวกัน

### โครงการระดับกลาง (3-4 ชั่วโมงต่อโครงการ)
4. **AI Toolkit Workflow**: ใช้ VS Code AI Toolkit เพื่อปรับแต่งและปรับใช้โมเดลตั้งแต่ต้นจนจบ
5. **Windows AI Foundry Application**: สร้างแอป Windows โดยใช้ Phi Silica API และการปรับแต่ง NPU
6. **Cross-Platform Deployment**: ปรับใช้โมเดลที่ปรับแต่งแล้วบน Windows (OpenVINO) และมือถือ (.NET MAUI)
7. **Function Calling Agent**: สร้าง AI agent ที่มีความสามารถในการเรียกฟังก์ชันสำหรับสถานการณ์ edge

### โครงการการบูรณาการขั้นสูง (4-5 ชั่วโมงต่อโครงการ)
8. **OpenVINO Optimization Pipeline**: ดำเนินการปรับแต่งโมเดลอย่างสมบูรณ์โดยใช้ NNCF และ GenAI toolkit
9. **SLMOps Pipeline**: ดำเนินการวงจรชีวิตโมเดลตั้งแต่การฝึกอบรมจนถึงการปรับใช้บน edge
10. **Multi-Model Edge System**: ปรับใช้โมเดลเฉพาะทางหลายตัวที่ทำงานร่วมกันบนฮาร์ดแวร์ edge
11. **MCP Integration System**: สร้างระบบ agentic โดยใช้ Model Context Protocol สำหรับการโต้ตอบกับเครื่องมือ
