# Sesión 4: Modelos de Última Generación – LLMs, SLMs e Inferencia en Dispositivos

## Resumen

Compara LLMs y SLMs, evalúa las ventajas y desventajas de la inferencia local frente a la nube, e implementa demostraciones que destaquen escenarios de EdgeAI utilizando Phi y ONNX Runtime. También exploraremos Chainlit RAG, opciones de inferencia con WebGPU y la integración de Open WebUI.

Referencias:
- Documentación de Foundry Local: https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/
- ONNX Runtime GenAI: https://onnxruntime.ai/
- Guía de Open WebUI (aplicación de chat con Open WebUI): https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/how-to/how-to-chat-application-with-open-web-ui

## Objetivos de Aprendizaje
- Comprender las diferencias entre LLM y SLM en términos de costo, latencia y precisión
- Elegir entre inferencia local y en la nube según las necesidades del negocio
- Implementar una demostración sencilla de RAG con Chainlit
- Explorar WebGPU para aceleración en el navegador
- Conectar Open WebUI a Foundry Local

## Parte 1: LLM vs SLM – Matriz de Decisión

Considera:
- Latencia: los SLMs en dispositivos suelen ofrecer respuestas en menos de un segundo
- Costo: la inferencia local reduce los costos en la nube
- Privacidad: los datos sensibles permanecen en el dispositivo
- Capacidad: los LLMs pueden superar a los SLMs en tareas complejas
- Fiabilidad: las estrategias híbridas reducen el riesgo de tiempo de inactividad

## Parte 2: Local vs Nube – Patrones Híbridos

- Local primero con respaldo en la nube para prompts grandes o complejos
- Nube primero con local para escenarios sensibles a la privacidad o sin conexión
- Enrutamiento por tipo de tarea (generación de código a DeepSeek, chat general a Phi/Qwen)

## Parte 3: Aplicación de Chat RAG con Chainlit (Mínima)

Instalar dependencias:
```powershell
python -m venv .venv
.\.venv\Scripts\activate
pip install chainlit requests
```

`app.py`:
```python
import chainlit as cl
import requests

BASE_URL = "http://localhost:8000"
MODEL = "phi-4-mini"

@cl.on_chat_start
async def start():
    await cl.Message(content="Welcome to Local RAG Chat!").send()

@cl.on_message
async def main(message: cl.Message):
    # Basic passthrough to Foundry Local (extend with retrieval)
    payload = {
        "model": MODEL,
        "messages": [{"role": "user", "content": message.content}],
        "max_tokens": 300
    }
    r = requests.post(f"{BASE_URL}/v1/chat/completions", json=payload, timeout=60)
    r.raise_for_status()
    content = r.json()["choices"][0]["message"]["content"]
    await cl.Message(content=content).send()
```

Ejecutar:
```powershell
# Ensure a model is running
foundry model run phi-4-mini

# Launch Chainlit
chainlit run app.py -w
```

Extender: agrega un recuperador sencillo (archivos locales) y antepone el contexto recuperado al prompt del usuario.

## Parte 4: Inferencia con WebGPU (Introducción)

Ejecuta modelos pequeños directamente en el navegador utilizando WebGPU. Esto es ideal para demostraciones centradas en la privacidad y experiencias sin instalación. A continuación, un ejemplo paso a paso utilizando ONNX Runtime Web con el proveedor de ejecución WebGPU.

1) Verifica soporte para WebGPU
- Navegadores Chromium: chrome://gpu → confirma que “WebGPU” está habilitado
- Verificación programática (también lo verificaremos en el código): `if (!('gpu' in navigator)) { /* no WebGPU */ }`

2) Crea un proyecto mínimo
Crea una carpeta y dos archivos: `index.html` y `main.js`.

`index.html`
```html
<!doctype html>
<html>
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <title>WebGPU + ONNX Runtime Web (Demo)</title>
    <!-- Load ONNX Runtime Web (WebGPU build) from CDN -->
    <script src="https://cdn.jsdelivr.net/npm/onnxruntime-web/dist/ort.webgpu.min.js"></script>
    <style>
      body { font-family: system-ui, sans-serif; margin: 2rem; }
      pre { background: #f5f5f5; padding: 1rem; overflow:auto; }
    </style>
  </head>
  <body>
    <h1>WebGPU + ONNX Runtime Web</h1>
    <p id="status">Initializing…</p>
    <pre id="output"></pre>
    <script type="module" src="./main.js"></script>
  </body>
</html>
```

`main.js`
```javascript
const statusEl = document.getElementById('status');
const outputEl = document.getElementById('output');

function log(msg) {
  outputEl.textContent += `${msg}\n`;
}

(async () => {
  try {
    if (!('gpu' in navigator)) {
      statusEl.textContent = 'WebGPU not available in this browser.';
      return;
    }
    statusEl.textContent = 'WebGPU detected. Loading model…';

    // Use a tiny identity model (or any small ONNX) hosted remotely.
    // For demo, we synthesize a trivial input and expect the same output shape.
    // Replace with your own model URL if desired.
    const modelUrl = 'https://huggingface.co/onnx/models/resolve/main/vision/classification/mnist-12/mnist-12.onnx?download=true';

    // Create a WebGPU session
    const session = await ort.InferenceSession.create(modelUrl, {
      executionProviders: ['webgpu']
    });
    log('Session created with WebGPU.');

    // Create a dummy input of the correct shape for MNIST (1x1x28x28)
    const inputData = new Float32Array(1 * 1 * 28 * 28);
    // Minimal non-zero to avoid NaNs in some models
    inputData[0] = 1.0;
    const input = new ort.Tensor('float32', inputData, [1, 1, 28, 28]);

    // The MNIST model input name is typically 'Input3' or similar; we can inspect session.inputNames
    const feeds = {};
    const inputName = session.inputNames[0];
    feeds[inputName] = input;

    const results = await session.run(feeds);
    const outputName = session.outputNames[0];
    const output = results[outputName]; // Float32Array logits

    statusEl.textContent = 'Inference complete.';
    log(`Input name: ${inputName}`);
    log(`Output name: ${outputName}`);
    log(`Output length: ${output.data.length}`);

    // Argmax to get predicted digit
    let bestIdx = 0; let bestVal = -Infinity;
    for (let i = 0; i < output.data.length; i++) {
      if (output.data[i] > bestVal) { bestVal = output.data[i]; bestIdx = i; }
    }
    log(`Predicted class (argmax): ${bestIdx}`);
  } catch (e) {
    statusEl.textContent = 'Error during inference.';
    log(e?.stack || String(e));
  }
})();
```

3) Servir localmente (Windows cmd.exe)
Usa un servidor estático simple para que el navegador pueda obtener el modelo.

```cmd
REM In cmd.exe
mkdir webgpu-demo
cd webgpu-demo
REM Save index.html and main.js here

REM Option A: Python (if installed)
py -m http.server 5173

REM Option B: Node serve (requires Node.js)
REM npx serve -p 5173 .
```

Abre http://localhost:5173 en tu navegador. Deberías ver registros de inicialización, creación de sesión con WebGPU y una predicción argmax.

4) Solución de problemas
- Si WebGPU no está disponible: actualiza Chrome/Edge y asegúrate de que los controladores de GPU estén actualizados, luego verifica chrome://flags para “Enable WebGPU”.
- Si ocurren errores de CORS o fetch: asegúrate de servir los archivos sobre http:// (no file://) y que la URL del modelo permita solicitudes de origen cruzado.
- Fallback a CPU: cambia `executionProviders: ['wasm']` para verificar el comportamiento básico.

5) Próximos pasos
- Sustituye por un modelo ONNX específico de dominio (por ejemplo, clasificación de imágenes o un modelo de texto pequeño).
- Agrega lógica de preprocesamiento/postprocesamiento para entradas reales.
- Para modelos más grandes o latencia en producción, prefiere Foundry Local o ONNX Runtime Server.

## Parte 5: Open WebUI + Foundry Local (Paso a paso)

Esto conecta Open WebUI al endpoint compatible con OpenAI de Foundry Local para una interfaz de chat local.

1) Requisitos previos
- Foundry Local instalado y funcionando (`foundry --version`)
- Un modelo listo para ejecutarse localmente (por ejemplo, `phi-4-mini`)
- Docker Desktop instalado (recomendado para Open WebUI)

2) Inicia un modelo con Foundry Local
```powershell
foundry model run phi-4-mini
```
Esto expone una API compatible con OpenAI en `http://localhost:8000`.

3) Inicia Open WebUI (Docker)
```powershell
# Pull and run Open WebUI
docker pull ghcr.io/open-webui/open-webui:main

docker run -d --name open-webui -p 3000:8080 ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
Notas:
- En Windows, `host.docker.internal` permite que el contenedor alcance tu host en `localhost`.
- Configuramos `OPENAI_API_BASE_URL` al endpoint de Foundry Local y una clave de API ficticia `OPENAI_API_KEY`.

4) Configurar desde la interfaz de Open WebUI (alternativa)
- Navega a http://localhost:3000
- Completa la configuración inicial (usuario administrador)
- Ve a Configuración → Modelos/Proveedores
- Establece URL Base: `http://host.docker.internal:8000/v1`
- Establece Clave API: `local-key` (placeholder)
- Guarda

5) Ejecuta un prompt de prueba
- En el chat de Open WebUI, selecciona o ingresa el nombre del modelo `phi-4-mini`
- Prompt: “Enumera cinco beneficios de la inferencia de IA en dispositivos.”
- Deberías ver una respuesta transmitida desde tu modelo local

6) Solución de problemas
```powershell
# Verify Foundry Local is serving and responding
curl http://localhost:8000/v1/models

# Check Docker logs for Open WebUI
docker logs -f open-webui

# If you get ECONNREFUSED, ensure the model is running and port 8000 is open
# If the model name doesn't appear, try typing it manually in Open WebUI
```

7) Opcional: Persistir datos de Open WebUI
```powershell
# Stop and remove the container
docker stop open-webui
docker rm open-webui

# Re-run with a volume to persist data
mkdir %USERPROFILE%\openwebui-data

docker run -d --name open-webui -p 3000:8080 ^
  -v %USERPROFILE%\openwebui-data:/app/backend/data ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
