# Сесія 4: Передові моделі – LLM, SLM та локальне виконання

## Огляд

Порівняйте LLM і SLM, оцініть компроміси між локальним і хмарним виконанням, а також реалізуйте демонстрації, які демонструють сценарії EdgeAI за допомогою Phi та ONNX Runtime. Ми також розглянемо Chainlit RAG, варіанти виконання WebGPU та інтеграцію Open WebUI.

Ресурси:
- Документація Foundry Local: https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/
- ONNX Runtime GenAI: https://onnxruntime.ai/
- Інструкція Open WebUI (чат-додаток з Open WebUI): https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/how-to/how-to-chat-application-with-open-web-ui

## Цілі навчання
- Зрозуміти компроміси між LLM і SLM щодо вартості, затримки та точності
- Вибрати між локальним і хмарним виконанням для конкретних бізнес-потреб
- Реалізувати невеликий RAG-додаток з Chainlit
- Дослідити WebGPU для прискорення в браузері
- Підключити Open WebUI до Foundry Local

## Частина 1: LLM vs SLM – Матриця рішень

Розгляньте:
- Затримка: SLM на пристрої часто забезпечують відповідь за частки секунди
- Вартість: локальне виконання знижує витрати на хмару
- Конфіденційність: чутливі дані залишаються на пристрої
- Можливості: LLM можуть перевершувати SLM у складних завданнях
- Надійність: гібридні стратегії зменшують ризик простою

## Частина 2: Локальне vs Хмарне – Гібридні шаблони

- Локальне виконання з хмарним резервом для великих/складних запитів
- Хмарне виконання з локальним для конфіденційних або офлайн сценаріїв
- Маршрутизація за типом завдання (генерація коду до DeepSeek, загальний чат до Phi/Qwen)

## Частина 3: RAG чат-додаток з Chainlit (Мінімальний)

Встановіть залежності:
```powershell
python -m venv .venv
.\.venv\Scripts\activate
pip install chainlit requests
```

`app.py`:
```python
import chainlit as cl
import requests

BASE_URL = "http://localhost:8000"
MODEL = "phi-4-mini"

@cl.on_chat_start
async def start():
    await cl.Message(content="Welcome to Local RAG Chat!").send()

@cl.on_message
async def main(message: cl.Message):
    # Basic passthrough to Foundry Local (extend with retrieval)
    payload = {
        "model": MODEL,
        "messages": [{"role": "user", "content": message.content}],
        "max_tokens": 300
    }
    r = requests.post(f"{BASE_URL}/v1/chat/completions", json=payload, timeout=60)
    r.raise_for_status()
    content = r.json()["choices"][0]["message"]["content"]
    await cl.Message(content=content).send()
```

Запустіть:
```powershell
# Ensure a model is running
foundry model run phi-4-mini

# Launch Chainlit
chainlit run app.py -w
```

Розширте: додайте простий пошуковик (локальні файли) і додайте знайдений контекст до запиту користувача.

## Частина 4: Виконання WebGPU (Попередження)

Запускайте невеликі моделі безпосередньо в браузері за допомогою WebGPU. Це ідеально для демонстрацій з пріоритетом конфіденційності та досвіду без встановлення. Нижче наведено мінімальний покроковий приклад використання ONNX Runtime Web з провайдером виконання WebGPU.

1) Перевірте підтримку WebGPU
- Браузери Chromium: chrome://gpu → підтвердьте, що “WebGPU” увімкнено
- Програмна перевірка (ми також перевіримо в коді): `if (!('gpu' in navigator)) { /* no WebGPU */ }`

2) Створіть мінімальний проєкт
Створіть папку та два файли: `index.html` і `main.js`.

`index.html`
```html
<!doctype html>
<html>
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <title>WebGPU + ONNX Runtime Web (Demo)</title>
    <!-- Load ONNX Runtime Web (WebGPU build) from CDN -->
    <script src="https://cdn.jsdelivr.net/npm/onnxruntime-web/dist/ort.webgpu.min.js"></script>
    <style>
      body { font-family: system-ui, sans-serif; margin: 2rem; }
      pre { background: #f5f5f5; padding: 1rem; overflow:auto; }
    </style>
  </head>
  <body>
    <h1>WebGPU + ONNX Runtime Web</h1>
    <p id="status">Initializing…</p>
    <pre id="output"></pre>
    <script type="module" src="./main.js"></script>
  </body>
</html>
```

`main.js`
```javascript
const statusEl = document.getElementById('status');
const outputEl = document.getElementById('output');

function log(msg) {
  outputEl.textContent += `${msg}\n`;
}

(async () => {
  try {
    if (!('gpu' in navigator)) {
      statusEl.textContent = 'WebGPU not available in this browser.';
      return;
    }
    statusEl.textContent = 'WebGPU detected. Loading model…';

    // Use a tiny identity model (or any small ONNX) hosted remotely.
    // For demo, we synthesize a trivial input and expect the same output shape.
    // Replace with your own model URL if desired.
    const modelUrl = 'https://huggingface.co/onnx/models/resolve/main/vision/classification/mnist-12/mnist-12.onnx?download=true';

    // Create a WebGPU session
    const session = await ort.InferenceSession.create(modelUrl, {
      executionProviders: ['webgpu']
    });
    log('Session created with WebGPU.');

    // Create a dummy input of the correct shape for MNIST (1x1x28x28)
    const inputData = new Float32Array(1 * 1 * 28 * 28);
    // Minimal non-zero to avoid NaNs in some models
    inputData[0] = 1.0;
    const input = new ort.Tensor('float32', inputData, [1, 1, 28, 28]);

    // The MNIST model input name is typically 'Input3' or similar; we can inspect session.inputNames
    const feeds = {};
    const inputName = session.inputNames[0];
    feeds[inputName] = input;

    const results = await session.run(feeds);
    const outputName = session.outputNames[0];
    const output = results[outputName]; // Float32Array logits

    statusEl.textContent = 'Inference complete.';
    log(`Input name: ${inputName}`);
    log(`Output name: ${outputName}`);
    log(`Output length: ${output.data.length}`);

    // Argmax to get predicted digit
    let bestIdx = 0; let bestVal = -Infinity;
    for (let i = 0; i < output.data.length; i++) {
      if (output.data[i] > bestVal) { bestVal = output.data[i]; bestIdx = i; }
    }
    log(`Predicted class (argmax): ${bestIdx}`);
  } catch (e) {
    statusEl.textContent = 'Error during inference.';
    log(e?.stack || String(e));
  }
})();
```

3) Запустіть локально (Windows cmd.exe)
Використовуйте простий статичний сервер, щоб браузер міг отримати модель.

```cmd
REM In cmd.exe
mkdir webgpu-demo
cd webgpu-demo
REM Save index.html and main.js here

REM Option A: Python (if installed)
py -m http.server 5173

REM Option B: Node serve (requires Node.js)
REM npx serve -p 5173 .
```

Відкрийте http://localhost:5173 у вашому браузері. Ви повинні побачити журнали ініціалізації, створення сесії з WebGPU та прогноз argmax.

4) Вирішення проблем
- Якщо WebGPU недоступний: оновіть Chrome/Edge і переконайтеся, що драйвери GPU актуальні, потім перевірте chrome://flags для “Enable WebGPU”.
- Якщо виникають помилки CORS або fetch: переконайтеся, що ви запускаєте файли через http:// (а не file://) і URL моделі дозволяє запити з іншого джерела.
- Резерв до CPU: змініть `executionProviders: ['wasm']`, щоб перевірити базову поведінку.

5) Наступні кроки
- Замініть на доменну ONNX модель (наприклад, класифікація зображень або невелика текстова модель).
- Додайте логіку попередньої/постобробки для реальних даних.
- Для більших моделей або виробничої затримки віддавайте перевагу Foundry Local або ONNX Runtime Server.

## Частина 5: Open WebUI + Foundry Local (Покроково)

Це підключає Open WebUI до сумісного з OpenAI кінцевого пункту Foundry Local для локального інтерфейсу чату.

1) Попередні умови
- Встановлений і працюючий Foundry Local (`foundry --version`)
- Одна модель готова до запуску локально (наприклад, `phi-4-mini`)
- Встановлений Docker Desktop (рекомендовано для Open WebUI)

2) Запустіть модель з Foundry Local
```powershell
foundry model run phi-4-mini
```
Це відкриває сумісний з OpenAI API на `http://localhost:8000`.

3) Запустіть Open WebUI (Docker)
```powershell
# Pull and run Open WebUI
docker pull ghcr.io/open-webui/open-webui:main

docker run -d --name open-webui -p 3000:8080 ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
Примітки:
- У Windows `host.docker.internal` дозволяє контейнеру звертатися до вашого хоста на `localhost`.
- Ми встановлюємо `OPENAI_API_BASE_URL` на кінцевий пункт Foundry Local і фіктивний `OPENAI_API_KEY`.

4) Налаштуйте через інтерфейс Open WebUI (альтернатива)
- Перейдіть до http://localhost:3000
- Завершіть початкове налаштування (адміністратор)
- Перейдіть до Налаштування → Моделі/Постачальники
- Встановіть базовий URL: `http://host.docker.internal:8000/v1`
- Встановіть API Key: `local-key` (заповнювач)
- Збережіть

5) Запустіть тестовий запит
- У чаті Open WebUI виберіть або введіть назву моделі `phi-4-mini`
- Запит: “Перелічіть п’ять переваг локального виконання AI.”
- Ви повинні побачити відповідь, передану з вашої локальної моделі

6) Вирішення проблем
```powershell
# Verify Foundry Local is serving and responding
curl http://localhost:8000/v1/models

# Check Docker logs for Open WebUI
docker logs -f open-webui

# If you get ECONNREFUSED, ensure the model is running and port 8000 is open
# If the model name doesn't appear, try typing it manually in Open WebUI
```

7) Додатково: Збереження даних Open WebUI
```powershell
# Stop and remove the container
docker stop open-webui
docker rm open-webui

# Re-run with a volume to persist data
mkdir %USERPROFILE%\openwebui-data

docker run -d --name open-webui -p 3000:8080 ^
  -v %USERPROFILE%\openwebui-data:/app/backend/data ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
