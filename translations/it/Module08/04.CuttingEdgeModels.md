# Sessione 4: Modelli all'avanguardia – LLM, SLM e inferenza su dispositivo

## Panoramica

Confronta LLM e SLM, valuta i compromessi tra inferenza locale e cloud, e implementa demo che mostrano scenari EdgeAI utilizzando Phi e ONNX Runtime. Metteremo anche in evidenza Chainlit RAG, opzioni di inferenza WebGPU e integrazione Open WebUI.

Riferimenti:
- Documentazione Foundry Local: https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/
- ONNX Runtime GenAI: https://onnxruntime.ai/
- Guida Open WebUI (app di chat con Open WebUI): https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/how-to/how-to-chat-application-with-open-web-ui

## Obiettivi di apprendimento
- Comprendere i compromessi tra LLM e SLM in termini di costo, latenza e accuratezza
- Scegliere tra inferenza locale e cloud per esigenze aziendali specifiche
- Implementare una piccola demo RAG con Chainlit
- Esplorare WebGPU per l'accelerazione lato browser
- Collegare Open WebUI a Foundry Local

## Parte 1: LLM vs SLM – Matrice decisionale

Considera:
- Latenza: gli SLM su dispositivo spesso offrono risposte in meno di un secondo
- Costo: l'inferenza locale riduce i costi del cloud
- Privacy: i dati sensibili rimangono sul dispositivo
- Capacità: gli LLM possono superare gli SLM in compiti complessi
- Affidabilità: strategie ibride riducono il rischio di downtime

## Parte 2: Locale vs Cloud – Modelli ibridi

- Locale come prima opzione con fallback al cloud per prompt grandi/complessi
- Cloud come prima opzione con locale per scenari sensibili alla privacy o offline
- Instradamento per tipo di attività (code-gen a DeepSeek, chat generale a Phi/Qwen)

## Parte 3: App di chat RAG con Chainlit (Minimale)

Installa le dipendenze:
```powershell
python -m venv .venv
.\.venv\Scripts\activate
pip install chainlit requests
```

`app.py`:
```python
import chainlit as cl
import requests

BASE_URL = "http://localhost:8000"
MODEL = "phi-4-mini"

@cl.on_chat_start
async def start():
    await cl.Message(content="Welcome to Local RAG Chat!").send()

@cl.on_message
async def main(message: cl.Message):
    # Basic passthrough to Foundry Local (extend with retrieval)
    payload = {
        "model": MODEL,
        "messages": [{"role": "user", "content": message.content}],
        "max_tokens": 300
    }
    r = requests.post(f"{BASE_URL}/v1/chat/completions", json=payload, timeout=60)
    r.raise_for_status()
    content = r.json()["choices"][0]["message"]["content"]
    await cl.Message(content=content).send()
```

Esegui:
```powershell
# Ensure a model is running
foundry model run phi-4-mini

# Launch Chainlit
chainlit run app.py -w
```

Estendi: aggiungi un semplice retriever (file locali) e anteponi il contesto recuperato al prompt dell'utente.

## Parte 4: Inferenza WebGPU (Introduzione)

Esegui modelli piccoli direttamente nel browser utilizzando WebGPU. Questo è ideale per demo orientate alla privacy e esperienze senza installazione. Di seguito un esempio minimo, passo dopo passo, utilizzando ONNX Runtime Web con il provider di esecuzione WebGPU.

1) Verifica il supporto WebGPU
- Browser Chromium: chrome://gpu → conferma che “WebGPU” sia abilitato
- Verifica programmatica (verificheremo anche nel codice): `if (!('gpu' in navigator)) { /* no WebGPU */ }`

2) Crea un progetto minimo
Crea una cartella e due file: `index.html` e `main.js`.

`index.html`
```html
<!doctype html>
<html>
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <title>WebGPU + ONNX Runtime Web (Demo)</title>
    <!-- Load ONNX Runtime Web (WebGPU build) from CDN -->
    <script src="https://cdn.jsdelivr.net/npm/onnxruntime-web/dist/ort.webgpu.min.js"></script>
    <style>
      body { font-family: system-ui, sans-serif; margin: 2rem; }
      pre { background: #f5f5f5; padding: 1rem; overflow:auto; }
    </style>
  </head>
  <body>
    <h1>WebGPU + ONNX Runtime Web</h1>
    <p id="status">Initializing…</p>
    <pre id="output"></pre>
    <script type="module" src="./main.js"></script>
  </body>
</html>
```

`main.js`
```javascript
const statusEl = document.getElementById('status');
const outputEl = document.getElementById('output');

function log(msg) {
  outputEl.textContent += `${msg}\n`;
}

(async () => {
  try {
    if (!('gpu' in navigator)) {
      statusEl.textContent = 'WebGPU not available in this browser.';
      return;
    }
    statusEl.textContent = 'WebGPU detected. Loading model…';

    // Use a tiny identity model (or any small ONNX) hosted remotely.
    // For demo, we synthesize a trivial input and expect the same output shape.
    // Replace with your own model URL if desired.
    const modelUrl = 'https://huggingface.co/onnx/models/resolve/main/vision/classification/mnist-12/mnist-12.onnx?download=true';

    // Create a WebGPU session
    const session = await ort.InferenceSession.create(modelUrl, {
      executionProviders: ['webgpu']
    });
    log('Session created with WebGPU.');

    // Create a dummy input of the correct shape for MNIST (1x1x28x28)
    const inputData = new Float32Array(1 * 1 * 28 * 28);
    // Minimal non-zero to avoid NaNs in some models
    inputData[0] = 1.0;
    const input = new ort.Tensor('float32', inputData, [1, 1, 28, 28]);

    // The MNIST model input name is typically 'Input3' or similar; we can inspect session.inputNames
    const feeds = {};
    const inputName = session.inputNames[0];
    feeds[inputName] = input;

    const results = await session.run(feeds);
    const outputName = session.outputNames[0];
    const output = results[outputName]; // Float32Array logits

    statusEl.textContent = 'Inference complete.';
    log(`Input name: ${inputName}`);
    log(`Output name: ${outputName}`);
    log(`Output length: ${output.data.length}`);

    // Argmax to get predicted digit
    let bestIdx = 0; let bestVal = -Infinity;
    for (let i = 0; i < output.data.length; i++) {
      if (output.data[i] > bestVal) { bestVal = output.data[i]; bestIdx = i; }
    }
    log(`Predicted class (argmax): ${bestIdx}`);
  } catch (e) {
    statusEl.textContent = 'Error during inference.';
    log(e?.stack || String(e));
  }
})();
```

3) Servi localmente (Windows cmd.exe)
Usa un semplice server statico affinché il browser possa recuperare il modello.

```cmd
REM In cmd.exe
mkdir webgpu-demo
cd webgpu-demo
REM Save index.html and main.js here

REM Option A: Python (if installed)
py -m http.server 5173

REM Option B: Node serve (requires Node.js)
REM npx serve -p 5173 .
```

Apri http://localhost:5173 nel tuo browser. Dovresti vedere i log di inizializzazione, la creazione della sessione con WebGPU e una previsione argmax.

4) Risoluzione dei problemi
- Se WebGPU non è disponibile: aggiorna Chrome/Edge e assicurati che i driver GPU siano aggiornati, poi controlla chrome://flags per “Enable WebGPU”.
- Se si verificano errori CORS o di fetch: assicurati di servire i file tramite http:// (non file://) e che l'URL del modello consenta richieste cross-origin.
- Fallback alla CPU: cambia `executionProviders: ['wasm']` per verificare il comportamento di base.

5) Prossimi passi
- Sostituisci con un modello ONNX specifico per il dominio (es. classificazione immagini o un piccolo modello di testo).
- Aggiungi logica di preprocessing/postprocessing per input reali.
- Per modelli più grandi o latenza in produzione, preferisci Foundry Local o ONNX Runtime Server.

## Parte 5: Open WebUI + Foundry Local (Passo dopo passo)

Questo collega Open WebUI all'endpoint compatibile con OpenAI di Foundry Local per un'interfaccia di chat locale.

1) Prerequisiti
- Foundry Local installato e funzionante (`foundry --version`)
- Un modello pronto per l'esecuzione locale (es. `phi-4-mini`)
- Docker Desktop installato (consigliato per Open WebUI)

2) Avvia un modello con Foundry Local
```powershell
foundry model run phi-4-mini
```
Questo espone un'API compatibile con OpenAI su `http://localhost:8000`.

3) Avvia Open WebUI (Docker)
```powershell
# Pull and run Open WebUI
docker pull ghcr.io/open-webui/open-webui:main

docker run -d --name open-webui -p 3000:8080 ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
Note:
- Su Windows, `host.docker.internal` consente al container di raggiungere il tuo host su `localhost`.
- Impostiamo `OPENAI_API_BASE_URL` sull'endpoint di Foundry Local e una chiave API fittizia `OPENAI_API_KEY`.

4) Configura dall'interfaccia Open WebUI (alternativa)
- Vai su http://localhost:3000
- Completa la configurazione iniziale (utente admin)
- Vai su Impostazioni → Modelli/Fornitori
- Imposta Base URL: `http://host.docker.internal:8000/v1`
- Imposta API Key: `local-key` (segnaposto)
- Salva

5) Esegui un prompt di test
- Nella chat di Open WebUI, seleziona o inserisci il nome del modello `phi-4-mini`
- Prompt: “Elenca cinque vantaggi dell'inferenza AI su dispositivo.”
- Dovresti vedere una risposta trasmessa dal tuo modello locale

6) Risoluzione dei problemi
```powershell
# Verify Foundry Local is serving and responding
curl http://localhost:8000/v1/models

# Check Docker logs for Open WebUI
docker logs -f open-webui

# If you get ECONNREFUSED, ensure the model is running and port 8000 is open
# If the model name doesn't appear, try typing it manually in Open WebUI
```

7) Opzionale: Persiste i dati di Open WebUI
```powershell
# Stop and remove the container
docker stop open-webui
docker rm open-webui

# Re-run with a volume to persist data
mkdir %USERPROFILE%\openwebui-data

docker run -d --name open-webui -p 3000:8080 ^
  -v %USERPROFILE%\openwebui-data:/app/backend/data ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
