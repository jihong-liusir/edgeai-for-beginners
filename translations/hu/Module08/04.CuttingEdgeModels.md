# 4. szekció: Legmodernebb modellek – LLM-ek, SLM-ek és eszközön történő inferencia

## Áttekintés

Hasonlítsuk össze az LLM-eket és az SLM-eket, értékeljük a helyi és felhőalapú inferencia közötti kompromisszumokat, és valósítsunk meg demókat, amelyek bemutatják az EdgeAI forgatókönyveket Phi és ONNX Runtime használatával. Kiemeljük továbbá a Chainlit RAG-t, a WebGPU inferencia lehetőségeit és az Open WebUI integrációt.

Hivatkozások:
- Foundry Local dokumentáció: https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/
- ONNX Runtime GenAI: https://onnxruntime.ai/
- Open WebUI útmutató (chat alkalmazás Open WebUI-val): https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/how-to/how-to-chat-application-with-open-web-ui

## Tanulási célok
- Értsd meg az LLM-ek és SLM-ek közötti kompromisszumokat költség, késleltetés és pontosság szempontjából
- Válassz helyi vagy felhőalapú inferenciát az üzleti igények alapján
- Valósíts meg egy egyszerű RAG demót Chainlit segítségével
- Fedezd fel a WebGPU-t böngészőoldali gyorsításra
- Kapcsold össze az Open WebUI-t a Foundry Local-lal

## 1. rész: LLM vs SLM – Döntési mátrix

Fontold meg:
- Késleltetés: Az SLM-ek eszközön gyakran szubszekundumos válaszokat adnak
- Költség: A helyi inferencia csökkenti a felhő költségeit
- Adatvédelem: Érzékeny adatok az eszközön maradnak
- Képesség: Az LLM-ek felülmúlhatják az SLM-eket összetett feladatokban
- Megbízhatóság: Hibrid stratégiák csökkentik a leállás kockázatát

## 2. rész: Helyi vs Felhő – Hibrid minták

- Helyi elsődleges, felhő tartalék nagy/összetett kérésekhez
- Felhő elsődleges, helyi adatvédelem-érzékeny vagy offline forgatókönyvekhez
- Feladat típus szerinti irányítás (kódgenerálás DeepSeek-hez, általános chat Phi/Qwen-hez)

## 3. rész: RAG chat alkalmazás Chainlit-tel (minimális)

Telepítsd a függőségeket:
```powershell
python -m venv .venv
.\.venv\Scripts\activate
pip install chainlit requests
```

`app.py`:
```python
import chainlit as cl
import requests

BASE_URL = "http://localhost:8000"
MODEL = "phi-4-mini"

@cl.on_chat_start
async def start():
    await cl.Message(content="Welcome to Local RAG Chat!").send()

@cl.on_message
async def main(message: cl.Message):
    # Basic passthrough to Foundry Local (extend with retrieval)
    payload = {
        "model": MODEL,
        "messages": [{"role": "user", "content": message.content}],
        "max_tokens": 300
    }
    r = requests.post(f"{BASE_URL}/v1/chat/completions", json=payload, timeout=60)
    r.raise_for_status()
    content = r.json()["choices"][0]["message"]["content"]
    await cl.Message(content=content).send()
```

Futtatás:
```powershell
# Ensure a model is running
foundry model run phi-4-mini

# Launch Chainlit
chainlit run app.py -w
```

Bővítés: adj hozzá egy egyszerű keresőt (helyi fájlok) és előzd meg a felhasználói kérést a visszakeresett kontextussal.

## 4. rész: WebGPU inferencia (Figyelemfelhívás)

Futtass kis modelleket közvetlenül a böngészőben WebGPU segítségével. Ez ideális adatvédelem-központú demókhoz és telepítés nélküli élményekhez. Az alábbiakban egy minimális, lépésről lépésre bemutatott példa található az ONNX Runtime Web használatával, WebGPU végrehajtási szolgáltatóval.

1) Ellenőrizd a WebGPU támogatást
- Chromium böngészők: chrome://gpu → erősítsd meg, hogy „WebGPU” engedélyezve van
- Programozott ellenőrzés (kódban is ellenőrizzük): `if (!('gpu' in navigator)) { /* nincs WebGPU */ }`

2) Hozz létre egy minimális projektet
Hozz létre egy mappát és két fájlt: `index.html` és `main.js`.

`index.html`
```html
<!doctype html>
<html>
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <title>WebGPU + ONNX Runtime Web (Demo)</title>
    <!-- Load ONNX Runtime Web (WebGPU build) from CDN -->
    <script src="https://cdn.jsdelivr.net/npm/onnxruntime-web/dist/ort.webgpu.min.js"></script>
    <style>
      body { font-family: system-ui, sans-serif; margin: 2rem; }
      pre { background: #f5f5f5; padding: 1rem; overflow:auto; }
    </style>
  </head>
  <body>
    <h1>WebGPU + ONNX Runtime Web</h1>
    <p id="status">Initializing…</p>
    <pre id="output"></pre>
    <script type="module" src="./main.js"></script>
  </body>
</html>
```

`main.js`
```javascript
const statusEl = document.getElementById('status');
const outputEl = document.getElementById('output');

function log(msg) {
  outputEl.textContent += `${msg}\n`;
}

(async () => {
  try {
    if (!('gpu' in navigator)) {
      statusEl.textContent = 'WebGPU not available in this browser.';
      return;
    }
    statusEl.textContent = 'WebGPU detected. Loading model…';

    // Use a tiny identity model (or any small ONNX) hosted remotely.
    // For demo, we synthesize a trivial input and expect the same output shape.
    // Replace with your own model URL if desired.
    const modelUrl = 'https://huggingface.co/onnx/models/resolve/main/vision/classification/mnist-12/mnist-12.onnx?download=true';

    // Create a WebGPU session
    const session = await ort.InferenceSession.create(modelUrl, {
      executionProviders: ['webgpu']
    });
    log('Session created with WebGPU.');

    // Create a dummy input of the correct shape for MNIST (1x1x28x28)
    const inputData = new Float32Array(1 * 1 * 28 * 28);
    // Minimal non-zero to avoid NaNs in some models
    inputData[0] = 1.0;
    const input = new ort.Tensor('float32', inputData, [1, 1, 28, 28]);

    // The MNIST model input name is typically 'Input3' or similar; we can inspect session.inputNames
    const feeds = {};
    const inputName = session.inputNames[0];
    feeds[inputName] = input;

    const results = await session.run(feeds);
    const outputName = session.outputNames[0];
    const output = results[outputName]; // Float32Array logits

    statusEl.textContent = 'Inference complete.';
    log(`Input name: ${inputName}`);
    log(`Output name: ${outputName}`);
    log(`Output length: ${output.data.length}`);

    // Argmax to get predicted digit
    let bestIdx = 0; let bestVal = -Infinity;
    for (let i = 0; i < output.data.length; i++) {
      if (output.data[i] > bestVal) { bestVal = output.data[i]; bestIdx = i; }
    }
    log(`Predicted class (argmax): ${bestIdx}`);
  } catch (e) {
    statusEl.textContent = 'Error during inference.';
    log(e?.stack || String(e));
  }
})();
```

3) Szolgáltatás helyben (Windows cmd.exe)
Használj egyszerű statikus szervert, hogy a böngésző elérje a modellt.

```cmd
REM In cmd.exe
mkdir webgpu-demo
cd webgpu-demo
REM Save index.html and main.js here

REM Option A: Python (if installed)
py -m http.server 5173

REM Option B: Node serve (requires Node.js)
REM npx serve -p 5173 .
```

Nyisd meg a http://localhost:5173 címet a böngésződben. Látnod kell az inicializációs naplókat, a WebGPU-val történő munkamenet létrehozását és egy argmax előrejelzést.

4) Hibakeresés
- Ha a WebGPU nem érhető el: frissítsd a Chrome/Edge böngészőt, győződj meg róla, hogy a GPU illesztőprogramok naprakészek, majd ellenőrizd a chrome://flags oldalon az „Enable WebGPU” beállítást.
- Ha CORS vagy fetch hibák lépnek fel: győződj meg róla, hogy a fájlokat http:// (nem file://) alatt szolgáltatod, és a modell URL-je engedélyezi a kereszt-eredetű kéréseket.
- Visszaállás CPU-ra: változtasd meg az `executionProviders: ['wasm']` beállítást az alapvető viselkedés ellenőrzéséhez.

5) Következő lépések
- Cseréld le egy domain-specifikus ONNX modellre (pl. képosztályozás vagy egy kis szövegmodell).
- Adj hozzá előfeldolgozási/utófeldolgozási logikát valódi bemenetekhez.
- Nagyobb modellekhez vagy termelési késleltetéshez részesítsd előnyben a Foundry Local-t vagy az ONNX Runtime Server-t.

## 5. rész: Open WebUI + Foundry Local (Lépésről lépésre)

Ez összekapcsolja az Open WebUI-t a Foundry Local OpenAI-kompatibilis végpontjával egy helyi chat UI-hoz.

1) Előfeltételek
- Foundry Local telepítve és működik (`foundry --version`)
- Egy modell készen áll a helyi futtatásra (pl. `phi-4-mini`)
- Docker Desktop telepítve (ajánlott az Open WebUI-hoz)

2) Indíts el egy modellt a Foundry Local-lal
```powershell
foundry model run phi-4-mini
```
Ez egy OpenAI-kompatibilis API-t tesz elérhetővé a `http://localhost:8000` címen.

3) Indítsd el az Open WebUI-t (Docker)
```powershell
# Pull and run Open WebUI
docker pull ghcr.io/open-webui/open-webui:main

docker run -d --name open-webui -p 3000:8080 ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
Megjegyzések:
- Windows alatt a `host.docker.internal` lehetővé teszi, hogy a konténer elérje a gazdagépet a `localhost` címen.
- Az `OPENAI_API_BASE_URL`-t a Foundry Local végpontjára állítjuk, és egy dummy `OPENAI_API_KEY`-t használunk.

4) Konfigurálás az Open WebUI UI-ból (alternatíva)
- Böngészd a http://localhost:3000 címet
- Végezd el a kezdeti beállítást (admin felhasználó)
- Menj a Beállítások → Modellek/Szolgáltatók menübe
- Állítsd be az alap URL-t: `http://host.docker.internal:8000/v1`
- Állítsd be az API kulcsot: `local-key` (helyettesítő)
- Mentés

5) Teszt kérések futtatása
- Az Open WebUI chatben válaszd ki vagy írd be a modell nevét: `phi-4-mini`
- Kérés: „Sorolj fel öt előnyt az eszközön történő AI inferenciával kapcsolatban.”
- Látnod kell egy választ, amelyet a helyi modell streamel.

6) Hibakeresés
```powershell
# Verify Foundry Local is serving and responding
curl http://localhost:8000/v1/models

# Check Docker logs for Open WebUI
docker logs -f open-webui

# If you get ECONNREFUSED, ensure the model is running and port 8000 is open
# If the model name doesn't appear, try typing it manually in Open WebUI
```

7) Opcionális: Open WebUI adatok megőrzése
```powershell
# Stop and remove the container
docker stop open-webui
docker rm open-webui

# Re-run with a volume to persist data
mkdir %USERPROFILE%\openwebui-data

docker run -d --name open-webui -p 3000:8080 ^
  -v %USERPROFILE%\openwebui-data:/app/backend/data ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
