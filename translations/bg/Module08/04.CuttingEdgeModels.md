# Сесия 4: Модели от последно поколение – LLMs, SLMs и локално извеждане

## Преглед

Сравнете LLMs и SLMs, оценете компромисите между локално и облачно извеждане и реализирайте демонстрации, които показват EdgeAI сценарии с Phi и ONNX Runtime. Ще разгледаме също Chainlit RAG, опции за извеждане с WebGPU и интеграция с Open WebUI.

Референции:
- Документация за Foundry Local: https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/
- ONNX Runtime GenAI: https://onnxruntime.ai/
- Ръководство за Open WebUI (чат приложение с Open WebUI): https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/how-to/how-to-chat-application-with-open-web-ui

## Цели на обучението
- Разберете компромисите между LLM и SLM по отношение на цена, латентност и точност
- Изберете между локално и облачно извеждане според конкретни бизнес нужди
- Реализирайте малка RAG демонстрация с Chainlit
- Изследвайте WebGPU за ускорение в браузъра
- Свържете Open WebUI с Foundry Local

## Част 1: LLM срещу SLM – Матрица за вземане на решения

Вземете предвид:
- Латентност: SLMs на устройството често предоставят отговори за под секунда
- Цена: локалното извеждане намалява разходите за облак
- Поверителност: чувствителните данни остават на устройството
- Способности: LLMs могат да превъзхождат SLMs при сложни задачи
- Надеждност: хибридните стратегии намаляват риска от прекъсвания

## Част 2: Локално срещу облачно – Хибридни модели

- Локално първо с облачен резерв за големи/сложни заявки
- Облачно първо с локално за чувствителни към поверителност или офлайн сценарии
- Насочване според типа задача (генериране на код към DeepSeek, общ чат към Phi/Qwen)

## Част 3: RAG чат приложение с Chainlit (Минимално)

Инсталирайте зависимости:
```powershell
python -m venv .venv
.\.venv\Scripts\activate
pip install chainlit requests
```

`app.py`:
```python
import chainlit as cl
import requests

BASE_URL = "http://localhost:8000"
MODEL = "phi-4-mini"

@cl.on_chat_start
async def start():
    await cl.Message(content="Welcome to Local RAG Chat!").send()

@cl.on_message
async def main(message: cl.Message):
    # Basic passthrough to Foundry Local (extend with retrieval)
    payload = {
        "model": MODEL,
        "messages": [{"role": "user", "content": message.content}],
        "max_tokens": 300
    }
    r = requests.post(f"{BASE_URL}/v1/chat/completions", json=payload, timeout=60)
    r.raise_for_status()
    content = r.json()["choices"][0]["message"]["content"]
    await cl.Message(content=content).send()
```

Стартирайте:
```powershell
# Ensure a model is running
foundry model run phi-4-mini

# Launch Chainlit
chainlit run app.py -w
```

Разширете: добавете прост извличащ механизъм (локални файлове) и добавете извлечения контекст към потребителската заявка.

## Част 4: Извеждане с WebGPU (Предварителна информация)

Стартирайте малки модели директно в браузъра с WebGPU. Това е идеално за демонстрации, които поставят поверителността на първо място, и за опит без инсталация. По-долу е минимален, стъпка по стъпка пример с ONNX Runtime Web и WebGPU изпълнителен доставчик.

1) Проверете поддръжката на WebGPU
- Браузъри Chromium: chrome://gpu → потвърдете, че “WebGPU” е активирано
- Програмна проверка (ще проверим и в кода): `if (!('gpu' in navigator)) { /* no WebGPU */ }`

2) Създайте минимален проект
Създайте папка и два файла: `index.html` и `main.js`.

`index.html`
```html
<!doctype html>
<html>
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <title>WebGPU + ONNX Runtime Web (Demo)</title>
    <!-- Load ONNX Runtime Web (WebGPU build) from CDN -->
    <script src="https://cdn.jsdelivr.net/npm/onnxruntime-web/dist/ort.webgpu.min.js"></script>
    <style>
      body { font-family: system-ui, sans-serif; margin: 2rem; }
      pre { background: #f5f5f5; padding: 1rem; overflow:auto; }
    </style>
  </head>
  <body>
    <h1>WebGPU + ONNX Runtime Web</h1>
    <p id="status">Initializing…</p>
    <pre id="output"></pre>
    <script type="module" src="./main.js"></script>
  </body>
</html>
```

`main.js`
```javascript
const statusEl = document.getElementById('status');
const outputEl = document.getElementById('output');

function log(msg) {
  outputEl.textContent += `${msg}\n`;
}

(async () => {
  try {
    if (!('gpu' in navigator)) {
      statusEl.textContent = 'WebGPU not available in this browser.';
      return;
    }
    statusEl.textContent = 'WebGPU detected. Loading model…';

    // Use a tiny identity model (or any small ONNX) hosted remotely.
    // For demo, we synthesize a trivial input and expect the same output shape.
    // Replace with your own model URL if desired.
    const modelUrl = 'https://huggingface.co/onnx/models/resolve/main/vision/classification/mnist-12/mnist-12.onnx?download=true';

    // Create a WebGPU session
    const session = await ort.InferenceSession.create(modelUrl, {
      executionProviders: ['webgpu']
    });
    log('Session created with WebGPU.');

    // Create a dummy input of the correct shape for MNIST (1x1x28x28)
    const inputData = new Float32Array(1 * 1 * 28 * 28);
    // Minimal non-zero to avoid NaNs in some models
    inputData[0] = 1.0;
    const input = new ort.Tensor('float32', inputData, [1, 1, 28, 28]);

    // The MNIST model input name is typically 'Input3' or similar; we can inspect session.inputNames
    const feeds = {};
    const inputName = session.inputNames[0];
    feeds[inputName] = input;

    const results = await session.run(feeds);
    const outputName = session.outputNames[0];
    const output = results[outputName]; // Float32Array logits

    statusEl.textContent = 'Inference complete.';
    log(`Input name: ${inputName}`);
    log(`Output name: ${outputName}`);
    log(`Output length: ${output.data.length}`);

    // Argmax to get predicted digit
    let bestIdx = 0; let bestVal = -Infinity;
    for (let i = 0; i < output.data.length; i++) {
      if (output.data[i] > bestVal) { bestVal = output.data[i]; bestIdx = i; }
    }
    log(`Predicted class (argmax): ${bestIdx}`);
  } catch (e) {
    statusEl.textContent = 'Error during inference.';
    log(e?.stack || String(e));
  }
})();
```

3) Стартирайте локално (Windows cmd.exe)
Използвайте прост статичен сървър, за да може браузърът да извлича модела.

```cmd
REM In cmd.exe
mkdir webgpu-demo
cd webgpu-demo
REM Save index.html and main.js here

REM Option A: Python (if installed)
py -m http.server 5173

REM Option B: Node serve (requires Node.js)
REM npx serve -p 5173 .
```

Отворете http://localhost:5173 в браузъра си. Трябва да видите логове за инициализация, създаване на сесия с WebGPU и предсказание с argmax.

4) Отстраняване на проблеми
- Ако WebGPU не е налично: актуализирайте Chrome/Edge и се уверете, че драйверите за GPU са актуални, след това проверете chrome://flags за “Enable WebGPU”.
- Ако възникнат грешки с CORS или извличане: уверете се, че файловете се обслужват през http:// (не file://) и че URL адресът на модела позволява заявки от други източници.
- Резерв към CPU: променете `executionProviders: ['wasm']`, за да проверите базовото поведение.

5) Следващи стъпки
- Заменете с домейн-специфичен ONNX модел (например за класификация на изображения или малък текстов модел).
- Добавете логика за предварителна/последваща обработка за реални входни данни.
- За по-големи модели или производствена латентност предпочитайте Foundry Local или ONNX Runtime Server.

## Част 5: Open WebUI + Foundry Local (Стъпка по стъпка)

Това свързва Open WebUI с OpenAI-съвместимия край на Foundry Local за локален чат интерфейс.

1) Предварителни изисквания
- Инсталиран и работещ Foundry Local (`foundry --version`)
- Един модел, готов за локално стартиране (например `phi-4-mini`)
- Инсталиран Docker Desktop (препоръчително за Open WebUI)

2) Стартирайте модел с Foundry Local
```powershell
foundry model run phi-4-mini
```
Това предоставя OpenAI-съвместим API на `http://localhost:8000`.

3) Стартирайте Open WebUI (Docker)
```powershell
# Pull and run Open WebUI
docker pull ghcr.io/open-webui/open-webui:main

docker run -d --name open-webui -p 3000:8080 ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
Бележки:
- В Windows, `host.docker.internal` позволява контейнерът да достигне вашия хост на `localhost`.
- Задаваме `OPENAI_API_BASE_URL` към крайната точка на Foundry Local и фиктивен `OPENAI_API_KEY`.

4) Конфигурирайте от интерфейса на Open WebUI (алтернатива)
- Отворете http://localhost:3000
- Завършете първоначалната настройка (администраторски потребител)
- Отидете на Настройки → Модели/Доставчици
- Задайте основен URL: `http://host.docker.internal:8000/v1`
- Задайте API ключ: `local-key` (заместител)
- Запазете

5) Стартирайте тестова заявка
- В чата на Open WebUI, изберете или въведете име на модел `phi-4-mini`
- Заявка: “Избройте пет предимства на AI извеждането на устройството.”
- Трябва да видите отговор, предаван от вашия локален модел

6) Отстраняване на проблеми
```powershell
# Verify Foundry Local is serving and responding
curl http://localhost:8000/v1/models

# Check Docker logs for Open WebUI
docker logs -f open-webui

# If you get ECONNREFUSED, ensure the model is running and port 8000 is open
# If the model name doesn't appear, try typing it manually in Open WebUI
```

7) По избор: Запазване на данни от Open WebUI
```powershell
# Stop and remove the container
docker stop open-webui
docker rm open-webui

# Re-run with a volume to persist data
mkdir %USERPROFILE%\openwebui-data

docker run -d --name open-webui -p 3000:8080 ^
  -v %USERPROFILE%\openwebui-data:/app/backend/data ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
