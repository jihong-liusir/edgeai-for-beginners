# Sesja 4: Najnowocześniejsze modele – LLM, SLM i inferencja na urządzeniu

## Przegląd

Porównaj LLM i SLM, oceń kompromisy między lokalną a chmurową inferencją oraz zaimplementuj demonstracje prezentujące scenariusze EdgeAI z użyciem Phi i ONNX Runtime. Podkreślimy również Chainlit RAG, opcje inferencji WebGPU oraz integrację Open WebUI.

Referencje:
- Dokumentacja Foundry Local: https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/
- ONNX Runtime GenAI: https://onnxruntime.ai/
- Instrukcja Open WebUI (aplikacja czatu z Open WebUI): https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/how-to/how-to-chat-application-with-open-web-ui

## Cele nauki
- Zrozumieć kompromisy między LLM a SLM w zakresie kosztów, opóźnień i dokładności
- Wybrać między lokalną a chmurową inferencją w zależności od potrzeb biznesowych
- Zaimplementować małą demonstrację RAG z Chainlit
- Zbadać WebGPU dla przyspieszenia po stronie przeglądarki
- Połączyć Open WebUI z Foundry Local

## Część 1: LLM vs SLM – Macierz decyzyjna

Rozważ:
- Opóźnienia: SLM na urządzeniu często zapewniają odpowiedzi w czasie poniżej sekundy
- Koszty: inferencja lokalna obniża koszty chmurowe
- Prywatność: wrażliwe dane pozostają na urządzeniu
- Możliwości: LLM mogą przewyższać SLM w złożonych zadaniach
- Niezawodność: strategie hybrydowe zmniejszają ryzyko przestojów

## Część 2: Lokalna vs chmurowa – Wzorce hybrydowe

- Podejście lokalne z chmurowym zapasowym dla dużych/złożonych zapytań
- Podejście chmurowe z lokalnym dla scenariuszy wymagających prywatności lub pracy offline
- Kierowanie według typu zadania (generowanie kodu do DeepSeek, ogólna rozmowa do Phi/Qwen)

## Część 3: Aplikacja czatu RAG z Chainlit (Minimalna)

Zainstaluj zależności:
```powershell
python -m venv .venv
.\.venv\Scripts\activate
pip install chainlit requests
```

`app.py`:
```python
import chainlit as cl
import requests

BASE_URL = "http://localhost:8000"
MODEL = "phi-4-mini"

@cl.on_chat_start
async def start():
    await cl.Message(content="Welcome to Local RAG Chat!").send()

@cl.on_message
async def main(message: cl.Message):
    # Basic passthrough to Foundry Local (extend with retrieval)
    payload = {
        "model": MODEL,
        "messages": [{"role": "user", "content": message.content}],
        "max_tokens": 300
    }
    r = requests.post(f"{BASE_URL}/v1/chat/completions", json=payload, timeout=60)
    r.raise_for_status()
    content = r.json()["choices"][0]["message"]["content"]
    await cl.Message(content=content).send()
```

Uruchom:
```powershell
# Ensure a model is running
foundry model run phi-4-mini

# Launch Chainlit
chainlit run app.py -w
```

Rozszerz: dodaj prosty mechanizm wyszukiwania (lokalne pliki) i poprzedź zapytanie użytkownika odnalezionym kontekstem.

## Część 4: Inferencja WebGPU (Zapowiedź)

Uruchamiaj małe modele bezpośrednio w przeglądarce za pomocą WebGPU. Idealne dla demonstracji z priorytetem prywatności i doświadczeń bez instalacji. Poniżej minimalny, krok po kroku przykład z użyciem ONNX Runtime Web z dostawcą wykonania WebGPU.

1) Sprawdź wsparcie WebGPU
- Przeglądarki Chromium: chrome://gpu → potwierdź, że „WebGPU” jest włączone
- Programatyczna kontrola (sprawdzimy również w kodzie): `if (!('gpu' in navigator)) { /* brak WebGPU */ }`

2) Utwórz minimalny projekt
Utwórz folder i dwa pliki: `index.html` oraz `main.js`.

`index.html`
```html
<!doctype html>
<html>
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <title>WebGPU + ONNX Runtime Web (Demo)</title>
    <!-- Load ONNX Runtime Web (WebGPU build) from CDN -->
    <script src="https://cdn.jsdelivr.net/npm/onnxruntime-web/dist/ort.webgpu.min.js"></script>
    <style>
      body { font-family: system-ui, sans-serif; margin: 2rem; }
      pre { background: #f5f5f5; padding: 1rem; overflow:auto; }
    </style>
  </head>
  <body>
    <h1>WebGPU + ONNX Runtime Web</h1>
    <p id="status">Initializing…</p>
    <pre id="output"></pre>
    <script type="module" src="./main.js"></script>
  </body>
</html>
```

`main.js`
```javascript
const statusEl = document.getElementById('status');
const outputEl = document.getElementById('output');

function log(msg) {
  outputEl.textContent += `${msg}\n`;
}

(async () => {
  try {
    if (!('gpu' in navigator)) {
      statusEl.textContent = 'WebGPU not available in this browser.';
      return;
    }
    statusEl.textContent = 'WebGPU detected. Loading model…';

    // Use a tiny identity model (or any small ONNX) hosted remotely.
    // For demo, we synthesize a trivial input and expect the same output shape.
    // Replace with your own model URL if desired.
    const modelUrl = 'https://huggingface.co/onnx/models/resolve/main/vision/classification/mnist-12/mnist-12.onnx?download=true';

    // Create a WebGPU session
    const session = await ort.InferenceSession.create(modelUrl, {
      executionProviders: ['webgpu']
    });
    log('Session created with WebGPU.');

    // Create a dummy input of the correct shape for MNIST (1x1x28x28)
    const inputData = new Float32Array(1 * 1 * 28 * 28);
    // Minimal non-zero to avoid NaNs in some models
    inputData[0] = 1.0;
    const input = new ort.Tensor('float32', inputData, [1, 1, 28, 28]);

    // The MNIST model input name is typically 'Input3' or similar; we can inspect session.inputNames
    const feeds = {};
    const inputName = session.inputNames[0];
    feeds[inputName] = input;

    const results = await session.run(feeds);
    const outputName = session.outputNames[0];
    const output = results[outputName]; // Float32Array logits

    statusEl.textContent = 'Inference complete.';
    log(`Input name: ${inputName}`);
    log(`Output name: ${outputName}`);
    log(`Output length: ${output.data.length}`);

    // Argmax to get predicted digit
    let bestIdx = 0; let bestVal = -Infinity;
    for (let i = 0; i < output.data.length; i++) {
      if (output.data[i] > bestVal) { bestVal = output.data[i]; bestIdx = i; }
    }
    log(`Predicted class (argmax): ${bestIdx}`);
  } catch (e) {
    statusEl.textContent = 'Error during inference.';
    log(e?.stack || String(e));
  }
})();
```

3) Serwuj lokalnie (Windows cmd.exe)
Użyj prostego serwera statycznego, aby przeglądarka mogła pobrać model.

```cmd
REM In cmd.exe
mkdir webgpu-demo
cd webgpu-demo
REM Save index.html and main.js here

REM Option A: Python (if installed)
py -m http.server 5173

REM Option B: Node serve (requires Node.js)
REM npx serve -p 5173 .
```

Otwórz http://localhost:5173 w przeglądarce. Powinieneś zobaczyć logi inicjalizacji, tworzenie sesji z WebGPU oraz predykcję argmax.

4) Rozwiązywanie problemów
- Jeśli WebGPU jest niedostępne: zaktualizuj Chrome/Edge i upewnij się, że sterowniki GPU są aktualne, następnie sprawdź chrome://flags dla „Enable WebGPU”.
- Jeśli występują błędy CORS lub fetch: upewnij się, że pliki są serwowane przez http:// (nie file://) i URL modelu pozwala na żądania cross-origin.
- Powrót do CPU: zmień `executionProviders: ['wasm']`, aby zweryfikować podstawowe działanie.

5) Kolejne kroki
- Zamień na model ONNX specyficzny dla domeny (np. klasyfikacja obrazów lub mały model tekstowy).
- Dodaj logikę przetwarzania wstępnego/końcowego dla rzeczywistych danych wejściowych.
- Dla większych modeli lub produkcyjnych opóźnień preferuj Foundry Local lub ONNX Runtime Server.

## Część 5: Open WebUI + Foundry Local (Krok po kroku)

To łączy Open WebUI z kompatybilnym z OpenAI punktem końcowym Foundry Local dla lokalnego interfejsu czatu.

1) Wymagania wstępne
- Zainstalowany i działający Foundry Local (`foundry --version`)
- Jeden model gotowy do uruchomienia lokalnie (np. `phi-4-mini`)
- Zainstalowany Docker Desktop (zalecane dla Open WebUI)

2) Uruchom model z Foundry Local
```powershell
foundry model run phi-4-mini
```
To udostępnia API kompatybilne z OpenAI pod adresem `http://localhost:8000`.

3) Uruchom Open WebUI (Docker)
```powershell
# Pull and run Open WebUI
docker pull ghcr.io/open-webui/open-webui:main

docker run -d --name open-webui -p 3000:8080 ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
Uwagi:
- Na Windows, `host.docker.internal` pozwala kontenerowi na dostęp do hosta pod `localhost`.
- Ustawiamy `OPENAI_API_BASE_URL` na punkt końcowy Foundry Local oraz fikcyjny `OPENAI_API_KEY`.

4) Konfiguracja z poziomu interfejsu Open WebUI (alternatywa)
- Przejdź do http://localhost:3000
- Ukończ początkową konfigurację (użytkownik admin)
- Przejdź do Ustawienia → Modele/Dostawcy
- Ustaw Base URL: `http://host.docker.internal:8000/v1`
- Ustaw API Key: `local-key` (symboliczny)
- Zapisz

5) Przetestuj zapytanie
- W czacie Open WebUI wybierz lub wpisz nazwę modelu `phi-4-mini`
- Zapytanie: „Wymień pięć korzyści z inferencji AI na urządzeniu.”
- Powinieneś zobaczyć odpowiedź przesyłaną strumieniowo z lokalnego modelu

6) Rozwiązywanie problemów
```powershell
# Verify Foundry Local is serving and responding
curl http://localhost:8000/v1/models

# Check Docker logs for Open WebUI
docker logs -f open-webui

# If you get ECONNREFUSED, ensure the model is running and port 8000 is open
# If the model name doesn't appear, try typing it manually in Open WebUI
```

7) Opcjonalnie: Zachowaj dane Open WebUI
```powershell
# Stop and remove the container
docker stop open-webui
docker rm open-webui

# Re-run with a volume to persist data
mkdir %USERPROFILE%\openwebui-data

docker run -d --name open-webui -p 3000:8080 ^
  -v %USERPROFILE%\openwebui-data:/app/backend/data ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
