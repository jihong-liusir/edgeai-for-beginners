# Session 4: Mga Makabagong Modelo – LLMs, SLMs, at On-Device Inference

## Pangkalahatang-ideya

Ihambing ang LLMs at SLMs, suriin ang mga trade-off sa pagitan ng lokal at cloud inference, at magpatupad ng mga demo na nagpapakita ng mga EdgeAI scenario gamit ang Phi at ONNX Runtime. I-highlight din natin ang Chainlit RAG, mga opsyon sa WebGPU inference, at integrasyon ng Open WebUI.

Mga Sanggunian:
- Foundry Local docs: https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/
- ONNX Runtime GenAI: https://onnxruntime.ai/
- Open WebUI how-to (chat app with Open WebUI): https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/how-to/how-to-chat-application-with-open-web-ui

## Mga Layunin sa Pagkatuto
- Maunawaan ang mga trade-off ng LLM vs SLM para sa gastos, latency, at katumpakan
- Pumili sa pagitan ng lokal at cloud inference para sa partikular na pangangailangan ng negosyo
- Magpatupad ng maliit na RAG demo gamit ang Chainlit
- Tuklasin ang WebGPU para sa browser-side acceleration
- Ikonekta ang Open WebUI sa Foundry Local

## Bahagi 1: LLM vs SLM – Decision Matrix

Isaalang-alang:
- Latency: Ang SLMs on-device ay madalas nagbibigay ng sub-second na tugon
- Gastos: Ang lokal na inference ay nakababawas sa gastos sa cloud
- Privacy: Ang sensitibong data ay nananatili sa device
- Kakayahan: Ang LLMs ay maaaring mas mahusay sa mas kumplikadong mga gawain
- Reliability: Ang hybrid na mga estratehiya ay nakababawas sa panganib ng downtime

## Bahagi 2: Lokal vs Cloud – Hybrid Patterns

- Lokal muna na may fallback sa cloud para sa malalaki/kumplikadong mga prompt
- Cloud muna na may lokal para sa privacy-sensitive o offline na mga sitwasyon
- Route ayon sa uri ng gawain (code-gen sa DeepSeek, general chat sa Phi/Qwen)

## Bahagi 3: RAG Chat App gamit ang Chainlit (Minimal)

I-install ang mga dependencies:
```powershell
python -m venv .venv
.\.venv\Scripts\activate
pip install chainlit requests
```

`app.py`:
```python
import chainlit as cl
import requests

BASE_URL = "http://localhost:8000"
MODEL = "phi-4-mini"

@cl.on_chat_start
async def start():
    await cl.Message(content="Welcome to Local RAG Chat!").send()

@cl.on_message
async def main(message: cl.Message):
    # Basic passthrough to Foundry Local (extend with retrieval)
    payload = {
        "model": MODEL,
        "messages": [{"role": "user", "content": message.content}],
        "max_tokens": 300
    }
    r = requests.post(f"{BASE_URL}/v1/chat/completions", json=payload, timeout=60)
    r.raise_for_status()
    content = r.json()["choices"][0]["message"]["content"]
    await cl.Message(content=content).send()
```

Patakbuhin:
```powershell
# Ensure a model is running
foundry model run phi-4-mini

# Launch Chainlit
chainlit run app.py -w
```

Palawakin: magdagdag ng simpleng retriever (lokal na mga file) at i-prepend ang nakuha na konteksto sa user prompt.

## Bahagi 4: WebGPU Inference (Heads-up)

Patakbuhin ang maliliit na modelo nang direkta sa browser gamit ang WebGPU. Ito ay perpekto para sa privacy-first na mga demo at zero-install na mga karanasan. Narito ang isang minimal, step-by-step na halimbawa gamit ang ONNX Runtime Web na may WebGPU execution provider.

1) Suriin ang suporta sa WebGPU
- Mga Chromium browser: chrome://gpu → kumpirmahin na “WebGPU” ay naka-enable
- Programmatic na pagsusuri (suriin din natin sa code): `if (!('gpu' in navigator)) { /* no WebGPU */ }`

2) Gumawa ng minimal na proyekto
Gumawa ng folder at dalawang file: `index.html` at `main.js`.

`index.html`
```html
<!doctype html>
<html>
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <title>WebGPU + ONNX Runtime Web (Demo)</title>
    <!-- Load ONNX Runtime Web (WebGPU build) from CDN -->
    <script src="https://cdn.jsdelivr.net/npm/onnxruntime-web/dist/ort.webgpu.min.js"></script>
    <style>
      body { font-family: system-ui, sans-serif; margin: 2rem; }
      pre { background: #f5f5f5; padding: 1rem; overflow:auto; }
    </style>
  </head>
  <body>
    <h1>WebGPU + ONNX Runtime Web</h1>
    <p id="status">Initializing…</p>
    <pre id="output"></pre>
    <script type="module" src="./main.js"></script>
  </body>
</html>
```

`main.js`
```javascript
const statusEl = document.getElementById('status');
const outputEl = document.getElementById('output');

function log(msg) {
  outputEl.textContent += `${msg}\n`;
}

(async () => {
  try {
    if (!('gpu' in navigator)) {
      statusEl.textContent = 'WebGPU not available in this browser.';
      return;
    }
    statusEl.textContent = 'WebGPU detected. Loading model…';

    // Use a tiny identity model (or any small ONNX) hosted remotely.
    // For demo, we synthesize a trivial input and expect the same output shape.
    // Replace with your own model URL if desired.
    const modelUrl = 'https://huggingface.co/onnx/models/resolve/main/vision/classification/mnist-12/mnist-12.onnx?download=true';

    // Create a WebGPU session
    const session = await ort.InferenceSession.create(modelUrl, {
      executionProviders: ['webgpu']
    });
    log('Session created with WebGPU.');

    // Create a dummy input of the correct shape for MNIST (1x1x28x28)
    const inputData = new Float32Array(1 * 1 * 28 * 28);
    // Minimal non-zero to avoid NaNs in some models
    inputData[0] = 1.0;
    const input = new ort.Tensor('float32', inputData, [1, 1, 28, 28]);

    // The MNIST model input name is typically 'Input3' or similar; we can inspect session.inputNames
    const feeds = {};
    const inputName = session.inputNames[0];
    feeds[inputName] = input;

    const results = await session.run(feeds);
    const outputName = session.outputNames[0];
    const output = results[outputName]; // Float32Array logits

    statusEl.textContent = 'Inference complete.';
    log(`Input name: ${inputName}`);
    log(`Output name: ${outputName}`);
    log(`Output length: ${output.data.length}`);

    // Argmax to get predicted digit
    let bestIdx = 0; let bestVal = -Infinity;
    for (let i = 0; i < output.data.length; i++) {
      if (output.data[i] > bestVal) { bestVal = output.data[i]; bestIdx = i; }
    }
    log(`Predicted class (argmax): ${bestIdx}`);
  } catch (e) {
    statusEl.textContent = 'Error during inference.';
    log(e?.stack || String(e));
  }
})();
```

3) I-serve nang lokal (Windows cmd.exe)
Gumamit ng simpleng static server para ma-fetch ng browser ang modelo.

```cmd
REM In cmd.exe
mkdir webgpu-demo
cd webgpu-demo
REM Save index.html and main.js here

REM Option A: Python (if installed)
py -m http.server 5173

REM Option B: Node serve (requires Node.js)
REM npx serve -p 5173 .
```

Buksan ang http://localhost:5173 sa iyong browser. Makikita mo ang mga initialization logs, session creation gamit ang WebGPU, at isang argmax prediction.

4) Pag-troubleshoot
- Kung ang WebGPU ay hindi magagamit: i-update ang Chrome/Edge at tiyakin na ang GPU drivers ay kasalukuyan, pagkatapos suriin ang chrome://flags para sa “Enable WebGPU”.
- Kung may mga CORS o fetch errors: tiyakin na ang mga file ay na-serve sa http:// (hindi file://) at ang model URL ay pinapayagan ang cross-origin requests.
- Fallback sa CPU: palitan ang `executionProviders: ['wasm']` para i-verify ang baseline behavior.

5) Mga susunod na hakbang
- Palitan ng domain-specific na ONNX model (hal., image classification o isang maliit na text model).
- Magdagdag ng preprocessing/postprocessing logic para sa totoong inputs.
- Para sa mas malalaking modelo o production latency, mas mainam ang Foundry Local o ONNX Runtime Server.

## Bahagi 5: Open WebUI + Foundry Local (Step-by-step)

Ikonekta ang Open WebUI sa OpenAI-compatible endpoint ng Foundry Local para sa lokal na chat UI.

1) Mga Kinakailangan
- Foundry Local na naka-install at gumagana (`foundry --version`)
- Isang model na handang patakbuhin nang lokal (hal., `phi-4-mini`)
- Docker Desktop na naka-install (inirerekomenda para sa Open WebUI)

2) Patakbuhin ang isang model gamit ang Foundry Local
```powershell
foundry model run phi-4-mini
```
Ito ay nag-e-expose ng OpenAI-compatible API sa `http://localhost:8000`.

3) Patakbuhin ang Open WebUI (Docker)
```powershell
# Pull and run Open WebUI
docker pull ghcr.io/open-webui/open-webui:main

docker run -d --name open-webui -p 3000:8080 ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
Mga Tala:
- Sa Windows, ang `host.docker.internal` ay nagpapahintulot sa container na maabot ang iyong host sa `localhost`.
- Itinakda natin ang `OPENAI_API_BASE_URL` sa endpoint ng Foundry Local at isang dummy na `OPENAI_API_KEY`.

4) I-configure mula sa Open WebUI UI (alternatibo)
- Buksan ang http://localhost:3000
- Kumpletuhin ang initial setup (admin user)
- Pumunta sa Settings → Models/Providers
- Itakda ang Base URL: `http://host.docker.internal:8000/v1`
- Itakda ang API Key: `local-key` (placeholder)
- I-save

5) Patakbuhin ang isang test prompt
- Sa Open WebUI chat, piliin o ilagay ang pangalan ng model na `phi-4-mini`
- Prompt: “Magbigay ng limang benepisyo ng on-device AI inference.”
- Makikita mo ang tugon na streamed mula sa iyong lokal na model

6) Pag-troubleshoot
```powershell
# Verify Foundry Local is serving and responding
curl http://localhost:8000/v1/models

# Check Docker logs for Open WebUI
docker logs -f open-webui

# If you get ECONNREFUSED, ensure the model is running and port 8000 is open
# If the model name doesn't appear, try typing it manually in Open WebUI
```

7) Opsyonal: I-persist ang Open WebUI data
```powershell
# Stop and remove the container
docker stop open-webui
docker rm open-webui

# Re-run with a volume to persist data
mkdir %USERPROFILE%\openwebui-data

docker run -d --name open-webui -p 3000:8080 ^
  -v %USERPROFILE%\openwebui-data:/app/backend/data ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
