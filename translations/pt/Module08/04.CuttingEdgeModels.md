# Sessão 4: Modelos de Última Geração – LLMs, SLMs e Inferência em Dispositivos

## Visão Geral

Compare LLMs e SLMs, avalie os trade-offs entre inferência local e na nuvem, e implemente demonstrações que destacam cenários de EdgeAI usando Phi e ONNX Runtime. Também abordaremos Chainlit RAG, opções de inferência com WebGPU e integração com Open WebUI.

Referências:
- Documentação Foundry Local: https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/
- ONNX Runtime GenAI: https://onnxruntime.ai/
- Guia Open WebUI (aplicação de chat com Open WebUI): https://learn.microsoft.com/en-us/azure/ai-foundry/foundry-local/how-to/how-to-chat-application-with-open-web-ui

## Objetivos de Aprendizagem
- Compreender os trade-offs entre LLM e SLM em termos de custo, latência e precisão
- Escolher entre inferência local e na nuvem para necessidades específicas de negócios
- Implementar uma pequena demonstração RAG com Chainlit
- Explorar o WebGPU para aceleração no navegador
- Conectar Open WebUI ao Foundry Local

## Parte 1: LLM vs SLM – Matriz de Decisão

Considere:
- Latência: SLMs em dispositivos geralmente oferecem respostas em menos de um segundo
- Custo: a inferência local reduz os custos na nuvem
- Privacidade: dados sensíveis permanecem no dispositivo
- Capacidade: LLMs podem superar SLMs em tarefas complexas
- Fiabilidade: estratégias híbridas reduzem o risco de indisponibilidade

## Parte 2: Local vs Nuvem – Padrões Híbridos

- Local primeiro com fallback na nuvem para prompts grandes/complexos
- Nuvem primeiro com local para cenários sensíveis à privacidade ou offline
- Roteamento por tipo de tarefa (geração de código para DeepSeek, chat geral para Phi/Qwen)

## Parte 3: Aplicação de Chat RAG com Chainlit (Minimalista)

Instalar dependências:
```powershell
python -m venv .venv
.\.venv\Scripts\activate
pip install chainlit requests
```

`app.py`:
```python
import chainlit as cl
import requests

BASE_URL = "http://localhost:8000"
MODEL = "phi-4-mini"

@cl.on_chat_start
async def start():
    await cl.Message(content="Welcome to Local RAG Chat!").send()

@cl.on_message
async def main(message: cl.Message):
    # Basic passthrough to Foundry Local (extend with retrieval)
    payload = {
        "model": MODEL,
        "messages": [{"role": "user", "content": message.content}],
        "max_tokens": 300
    }
    r = requests.post(f"{BASE_URL}/v1/chat/completions", json=payload, timeout=60)
    r.raise_for_status()
    content = r.json()["choices"][0]["message"]["content"]
    await cl.Message(content=content).send()
```

Executar:
```powershell
# Ensure a model is running
foundry model run phi-4-mini

# Launch Chainlit
chainlit run app.py -w
```

Expandir: adicionar um simples recuperador (ficheiros locais) e pré-pendurar o contexto recuperado ao prompt do utilizador.

## Parte 4: Inferência com WebGPU (Introdução)

Execute modelos pequenos diretamente no navegador usando WebGPU. Isto é ideal para demonstrações focadas na privacidade e experiências sem instalação. Abaixo está um exemplo minimalista, passo a passo, usando ONNX Runtime Web com o fornecedor de execução WebGPU.

1) Verificar suporte ao WebGPU
- Navegadores Chromium: chrome://gpu → confirmar “WebGPU” ativado
- Verificação programática (também verificaremos no código): `if (!('gpu' in navigator)) { /* no WebGPU */ }`

2) Criar um projeto minimalista
Crie uma pasta e dois ficheiros: `index.html` e `main.js`.

`index.html`
```html
<!doctype html>
<html>
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <title>WebGPU + ONNX Runtime Web (Demo)</title>
    <!-- Load ONNX Runtime Web (WebGPU build) from CDN -->
    <script src="https://cdn.jsdelivr.net/npm/onnxruntime-web/dist/ort.webgpu.min.js"></script>
    <style>
      body { font-family: system-ui, sans-serif; margin: 2rem; }
      pre { background: #f5f5f5; padding: 1rem; overflow:auto; }
    </style>
  </head>
  <body>
    <h1>WebGPU + ONNX Runtime Web</h1>
    <p id="status">Initializing…</p>
    <pre id="output"></pre>
    <script type="module" src="./main.js"></script>
  </body>
</html>
```

`main.js`
```javascript
const statusEl = document.getElementById('status');
const outputEl = document.getElementById('output');

function log(msg) {
  outputEl.textContent += `${msg}\n`;
}

(async () => {
  try {
    if (!('gpu' in navigator)) {
      statusEl.textContent = 'WebGPU not available in this browser.';
      return;
    }
    statusEl.textContent = 'WebGPU detected. Loading model…';

    // Use a tiny identity model (or any small ONNX) hosted remotely.
    // For demo, we synthesize a trivial input and expect the same output shape.
    // Replace with your own model URL if desired.
    const modelUrl = 'https://huggingface.co/onnx/models/resolve/main/vision/classification/mnist-12/mnist-12.onnx?download=true';

    // Create a WebGPU session
    const session = await ort.InferenceSession.create(modelUrl, {
      executionProviders: ['webgpu']
    });
    log('Session created with WebGPU.');

    // Create a dummy input of the correct shape for MNIST (1x1x28x28)
    const inputData = new Float32Array(1 * 1 * 28 * 28);
    // Minimal non-zero to avoid NaNs in some models
    inputData[0] = 1.0;
    const input = new ort.Tensor('float32', inputData, [1, 1, 28, 28]);

    // The MNIST model input name is typically 'Input3' or similar; we can inspect session.inputNames
    const feeds = {};
    const inputName = session.inputNames[0];
    feeds[inputName] = input;

    const results = await session.run(feeds);
    const outputName = session.outputNames[0];
    const output = results[outputName]; // Float32Array logits

    statusEl.textContent = 'Inference complete.';
    log(`Input name: ${inputName}`);
    log(`Output name: ${outputName}`);
    log(`Output length: ${output.data.length}`);

    // Argmax to get predicted digit
    let bestIdx = 0; let bestVal = -Infinity;
    for (let i = 0; i < output.data.length; i++) {
      if (output.data[i] > bestVal) { bestVal = output.data[i]; bestIdx = i; }
    }
    log(`Predicted class (argmax): ${bestIdx}`);
  } catch (e) {
    statusEl.textContent = 'Error during inference.';
    log(e?.stack || String(e));
  }
})();
```

3) Servir localmente (Windows cmd.exe)
Use um servidor estático simples para que o navegador possa buscar o modelo.

```cmd
REM In cmd.exe
mkdir webgpu-demo
cd webgpu-demo
REM Save index.html and main.js here

REM Option A: Python (if installed)
py -m http.server 5173

REM Option B: Node serve (requires Node.js)
REM npx serve -p 5173 .
```

Abra http://localhost:5173 no seu navegador. Deverá ver os logs de inicialização, criação de sessão com WebGPU e uma previsão argmax.

4) Resolução de problemas
- Se o WebGPU estiver indisponível: atualize Chrome/Edge e certifique-se de que os drivers da GPU estão atualizados, depois verifique chrome://flags para “Enable WebGPU”.
- Se ocorrerem erros de CORS ou fetch: certifique-se de que está a servir os ficheiros via http:// (não file://) e que o URL do modelo permite pedidos de origem cruzada.
- Fallback para CPU: altere `executionProviders: ['wasm']` para verificar o comportamento básico.

5) Próximos passos

- Adicione lógica de pré-processamento/pós-processamento para entradas reais.
- Para modelos maiores ou latência em produção, prefira Foundry Local ou ONNX Runtime Server.

## Parte 5: Open WebUI + Foundry Local (Passo a passo)

Isto conecta o Open WebUI ao endpoint compatível com OpenAI do Foundry Local para uma interface de chat local.

1) Pré-requisitos
- Foundry Local instalado e funcional (`foundry --version`)
- Um modelo pronto para execução local (ex.: `phi-4-mini`)
- Docker Desktop instalado (recomendado para Open WebUI)

2) Iniciar um modelo com Foundry Local
```powershell
foundry model run phi-4-mini
```
Isto expõe uma API compatível com OpenAI em `http://localhost:8000`.

3) Iniciar Open WebUI (Docker)
```powershell
# Pull and run Open WebUI
docker pull ghcr.io/open-webui/open-webui:main

docker run -d --name open-webui -p 3000:8080 ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
Notas:
- No Windows, `host.docker.internal` permite que o container aceda ao seu host em `localhost`.
- Definimos `OPENAI_API_BASE_URL` para o endpoint do Foundry Local e uma chave de API fictícia `OPENAI_API_KEY`.

4) Configurar a partir da interface do Open WebUI (alternativa)
- Navegue para http://localhost:3000
- Complete a configuração inicial (utilizador administrador)
- Vá para Configurações → Modelos/Provedores
- Defina Base URL: `http://host.docker.internal:8000/v1`
- Defina API Key: `local-key` (placeholder)
- Guarde

5) Executar um prompt de teste
- No chat do Open WebUI, selecione ou insira o nome do modelo `phi-4-mini`
- Prompt: “Liste cinco benefícios da inferência de IA em dispositivos.”
- Deverá ver uma resposta transmitida pelo seu modelo local

6) Resolução de problemas
```powershell
# Verify Foundry Local is serving and responding
curl http://localhost:8000/v1/models

# Check Docker logs for Open WebUI
docker logs -f open-webui

# If you get ECONNREFUSED, ensure the model is running and port 8000 is open
# If the model name doesn't appear, try typing it manually in Open WebUI
```

7) Opcional: Persistir dados do Open WebUI
```powershell
# Stop and remove the container
docker stop open-webui
docker rm open-webui

# Re-run with a volume to persist data
mkdir %USERPROFILE%\openwebui-data

docker run -d --name open-webui -p 3000:8080 ^
  -v %USERPROFILE%\openwebui-data:/app/backend/data ^
  -e OPENAI_API_BASE_URL=http://host.docker.internal:8000/v1 ^
  -e OPENAI_API_KEY=local-key ^
  ghcr.io/open-webui/open-webui:main
```
